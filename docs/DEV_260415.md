# SDK 스트리밍 중복 응답 버그 수정

## 증상

요약 요청(이미지 분석 포함) 시 동일/유사한 응답이 2~4회 중복 출력되는 현상.
매번 발생하는 것은 아니고 간헐적으로 발생.

예시: RAG 아키텍처 이미지 분석 요청 시 "Classic RAG / Graph RAG / Agentic RAG" 요약이
약간씩 다른 도입부로 3회 연속 스트리밍됨.

---

## 원인 분석

### 실행 흐름

```
generator.ts (SDK path)
  └─ while (sdkAttempt < MAX_KEY_RETRIES)
       ├─ generateContentStream() 호출
       │    └─ for await (chunk of sdkStream)
       │         └─ sendEvent({ text: chunk })  ← 클라이언트에 실시간 전송
       │
       ├─ 스트림 도중 or 완료 직후 429/503 에러 발생
       │
       └─ catch: isRateLimit || isUnavailable → sdkAttempt++; continue
                 ↑
                 새 API 키로 스트림 재시작
                 → 전체 텍스트 다시 sendEvent ← 중복 발생
```

### 핵심 원인

`generateContentStream` 스트리밍 도중 또는 직후 429(RESOURCE_EXHAUSTED) / 503(UNAVAILABLE)
에러가 발생하면, 이미 `sendEvent`로 클라이언트에 전송된 텍스트가 있어도
catch 블록이 다음 API 키로 재시도(`sdkAttempt++; continue`).

재시도 시 새 스트림이 처음부터 전송되어 클라이언트에 중복 텍스트 누적.

`responseText`는 `try` 블록 내에서 선언되므로 각 시도마다 리셋되지만,
`sendEvent` 호출은 이미 일어난 뒤이므로 클라이언트 측에서는 중복으로 수신.

## 수정 내용

**파일**: `api/_lib/agent/nodes/generator.ts`

catch 블록 최상단에 `responseText` 존재 여부를 확인하는 guard 추가.
텍스트가 이미 전송됐으면 즉시 반환하여 재시도 자체를 막음.

```typescript
// 수정 전
} catch (err: any) {
    const isRateLimit = ...;
    const isUnavailable = ...;
    if (isRateLimit || isUnavailable) {
        sdkAttempt++;
        continue;  // 이미 텍스트가 전송됐어도 재시도
    }
    break;
}

// 수정 후
} catch (err: any) {
    // If text was already streamed to the client, do NOT retry — would cause duplicate output
    if (responseText) {
        console.warn('[LangGraph] Error after partial stream (err:', err?.status, ') — returning partial response to avoid duplication');
        sdkSuccess = true;
        return { messages: [new AIMessage(responseText)], groundingSources };
    }
    const isRateLimit = ...;
    const isUnavailable = ...;
    if (isRateLimit || isUnavailable) {
        sdkAttempt++;
        continue;
    }
    break;
}
```

## 수정 후 시나리오

| 에러 발생 시점 | 수정 전 | 수정 후 |
|---|---|---|
| 첫 청크 이전 (`responseText = ""`) | 재시도 ✅ | 재시도 ✅ (변화 없음) |
| 스트림 도중 (`responseText` 일부) | 재시도 → 중복 ❌ | 부분 텍스트 반환 ⚠️ |
| 스트림 완료 직후 (`responseText` 전체) | 재시도 → 중복 ❌ | 전체 텍스트 반환 ✅ |

**트레이드오프**: 스트림 중간 에러 시 응답이 잘릴 수 있음.
단, 스트림 중간 429는 드문 케이스이며, 중복 응답보다 부분 응답이 UX상 낫다고 판단.
첫 청크 이전 429는 가장 흔한 패턴으로 기존대로 retry 동작함.

## 진단된 문제 목록

### 🔴 P1 — Critical

#### #1 · 401 처리 누락 (`generator.ts`)

**현상**: SDK path catch에서 `isRateLimit`, `isUnavailable` 모두 false → `break` → LangChain path로 silent fallthrough.  
LangChain path의 401은 `throw err` → `chat.ts` 외부 catch → **`error.message` 전체가 SSE로 클라이언트에 노출**.  
키 자체는 invalid 상태로 마킹되지 않아 다음 요청에서도 동일한 키로 시도됨.

```ts
// generator.ts catch (현재)
} catch (err: any) {
    const isRateLimit = err?.status === 429 || ...;
    const isUnavailable = err?.status === 503 || ...;
    if (isRateLimit || isUnavailable) { ... continue; }
    break; // ← 401도 여기로. 키 무효화 없음.
}
```

**필요한 처리**:
- 401 감지 → 해당 키를 영구 또는 장기 비활성화
- `break` 후 다음 키가 있으면 retry 시도
- 클라이언트 노출 에러 메시지 sanitize

---

### 🟠 P2 — High

#### #2 · SDK → LangChain silent fallthrough (`generator.ts`)

**현상**: non-drug intent(general, biology 등)에서 SDK path가 전체 실패(모든 키 429 소진 또는 비-retryable 에러)할 경우, `sdkSuccess = false`인 채로 코드가 LangChain path로 진행됨. `allTools = []`로 invoke되어 동작은 하지만:
- 스트리밍 없음 (LangChain `invoke`는 non-streaming)
- 의도된 fallback인지 버그인지 코드 구조상 불명확
- 실패 원인이 로그에만 남고 모니터링 불가

```ts
if (!useLangChain) {
    while (...) { /* SDK 실패 */ }
    // sdkSuccess = false인 채로 여기 도달
}
// LangChain path가 아무 표시 없이 실행됨
let lcApiKey = apiKey;
```

**필요한 처리**:
- fallthrough 발생 시 명시적 경고 로그
- non-drug intent의 LangChain fallback이 의도된 동작인지 명확히 주석으로 문서화
- 또는 `sdkSuccess = false` 시 명시적 에러 처리

#### #3 · 전체 API 키 소진 시 순환 429 (`config.ts`)

**현상**: 모든 키가 `rateLimitedUntil` cooldown 중일 때:
```ts
// 현재 "last resort" 처리
return API_KEYS[currentKeyIndex]; // 이미 제한된 키 반환
```
이 키로 요청 → 또 429 → `markKeyRateLimited` → cooldown 갱신 → 같은 키로 계속 순환.  
또한 `currentKeyIndex`가 module-level 변수라 Fluid Compute 인스턴스 재사용 시 동시 요청 간 race condition 가능성.

**필요한 처리**:
- 모든 키 소진 시 `null` 반환 또는 최소 남은 cooldown 시간 계산 후 대기
- `currentKeyIndex` 접근을 atomic하게 처리 (또는 요청별 random 선택)

## 1라운드 적용 완료 (2026-04-15)

### 수정 1 — `config.ts` 전체 키 소진 시 null 반환

모든 키가 rate-limited 상태일 때 제한된 키를 재반환하던 "last resort" 로직 제거.
`null` 반환으로 변경 — 모든 호출자(`generator`, `router`, `vision`, `summarize-title`, `speech`)가 이미 null 처리 중이므로 사이드 이펙트 없음.

```ts
// 수정 전
return API_KEYS[currentKeyIndex]; // 이미 제한된 키 재반환 → 순환 429

// 수정 후
return null; // 호출자가 gracefully 처리
```

### 수정 2 — `router.ts` 429 시 `markKeyRateLimited`

Router LLM(gemini-2.5-flash-lite) 429 발생 시 해당 키를 60s cooldown으로 마킹.
이후 Generator가 같은 키를 즉시 재사용하는 연속 429 방지.

```ts
} catch (error: any) {
    const isRateLimit = error?.status === 429 || error?.message?.includes('RESOURCE_EXHAUSTED');
    if (isRateLimit && apiKey) markKeyRateLimited(apiKey);
    intent = heuristicCheck();
}
```

### 수정 3 — `summarize-title.ts`, `speech.ts` 429 시 `markKeyRateLimited`

두 엔드포인트 모두 catch에서 429 감지 후 `markKeyRateLimited` 추가.
세션 제목 생성, TTS 중 429가 발생해도 공유 키 풀에 반영됨.

### 수정 4 — `chat.ts` 에러 메시지 sanitize

내부 에러 메시지(`error.message`) 클라이언트 노출 제거.
에러 코드별 사용자 친화적 메시지로 분기.

```ts
const clientMessage =
  status === 429 || ...RESOURCE_EXHAUSTED
    ? '요청이 많아 잠시 지연되고 있습니다. 잠시 후 다시 시도해주세요.'
  : status === 503 || ...UNAVAILABLE
    ? '서버가 일시적으로 불안정합니다. 잠시 후 다시 시도해주세요.'
  : status === 401 || status === 403
    ? '인증 오류가 발생했습니다. 관리자에게 문의해주세요.'
  : '응답 생성 중 문제가 발생했습니다. 다시 시도해주세요.';
sendEvent({ error: clientMessage });
```

### 수정 5 — `useChatStream.ts` 에러 표시 방식 + 로딩 상태 고착 버그 수정

에러를 `setLoadingStatus`(로딩 바, 5초 후 소멸)로 표시하던 것을 `onError`(토스트)로 변경.
동시에 `finally`의 `if (!hasError)` 조건 제거 — 에러 발생 시 로딩 상태('이미지 분석 중...' 등)가
영구 고착되던 숨겨진 버그 수정.

```ts
// 수정 전
} catch (error: any) {
  hasError = true;
  setLoadingStatus(error.message);          // 로딩 바에 5초 표시
  setTimeout(() => setLoadingStatus(null), 5000);
} finally {
  setIsTyping(false);
  if (!hasError) setLoadingStatus(null);    // hasError=true라서 null 안 함 → 고착
}

// 수정 후
} catch (error: any) {
  hasError = true;
  onError(error.message);                   // 토스트로 표시
} finally {
  setIsTyping(false);
  setLoadingStatus(null);                   // 항상 클리어
}
```

## 2라운드 적용 완료 (2026-04-15)

### 수정 1 — `config.ts` `markKeyInvalid` 추가

401/403 응답 시 해당 키를 24시간 비활성화하는 함수 추가.
기존 `markKeyRateLimited`(60초)와 분리하여 에러 종류별 cooldown 차등화.

```ts
export const markKeyInvalid = (apiKey: string) => {
    rateLimitedUntil.set(apiKey, Date.now() + 24 * 60 * 60_000); // 24h
};
```

### 수정 2 — `generator.ts` 401/403 처리 + fallthrough 명시화

**SDK path**: `isAuth` 감지 → `markKeyInvalid` + 다음 키로 retry.
**LangChain path**: 401/403 re-throw 전 `markKeyInvalid` 호출.
**SDK fallthrough**: `!useLangChain && !sdkSuccess` 시 명시적 경고 로그.
**lcApiKey 개선**: `getNextApiKey() ?? apiKey` — SDK에서 소진된 키 재사용 최소화.

```ts
// SDK catch
const isAuth = err?.status === 401 || err?.status === 403;
if (isAuth) {
    markKeyInvalid(sdkApiKey);
    const nextKey = getNextApiKey();
    if (nextKey && nextKey !== sdkApiKey) { sdkApiKey = nextKey; continue; }
}

// LangChain catch (re-throw 전)
const isAuth = err?.status === 401 || err?.status === 403;
if (isAuth) markKeyInvalid(lcApiKey);
throw err;
```

### 수정 3 — `vision.ts` 429 retry + JSON parse 로그

**retry**: `MAX_ATTEMPTS = Math.min(2, API_KEYS.length)` — 429 발생 시 다음 키로 1회 재시도.
message 객체는 루프 바깥에서 1회 생성 (성능 중립).
**JSON parse 로그**: `console.warn` 추가로 파싱 실패 원인 진단 가능.

### 수정 4 — `fetch-transcript.ts` AbortController timeout

YouTube 페이지 fetch: 10초, 트랜스크립트 XML fetch: 8초.
타임아웃 시 `fetchYoutubeTranscript` → `null` 반환 → useChatStream 직접 비디오 분석 fallback.

## 4라운드 적용 완료 (2026-04-16)

### 수정 1 — `generator.ts` `responseText` 스코프 버그 수정

**발견**: `let responseText = ""` / `let groundingSources: any[] = []`가 `try {}` 블록 내에서 선언되어 있었고, 중복 응답 방지 가드(`if (responseText) return immediately`)가 sibling `catch {}` 블록에서 이를 참조하고 있었음.

**영향**: `target: "ESNext"` + `strict: true` 환경에서 블록 스코프(`let`)는 `try`와 `catch`를 별개의 스코프로 분리. 런타임에서 `ReferenceError: responseText is not defined`가 발생하여 실제로 중복 응답 가드가 동작하지 않을 수 있었음.

**수정**: 두 변수를 `while` 루프 안, `try {}` 바깥으로 이동. 이제 `catch {}` 블록에서 정상 접근 가능.

```ts
// 수정 전
while (...) {
    try {
        let responseText = "";       // ← try 블록 내 선언
        let groundingSources = [];
        ...
    } catch (err) {
        if (responseText) { ... }    // ← ReferenceError 발생
    }
}

// 수정 후
while (...) {
    let responseText = "";           // ← while 루프 직하 선언
    let groundingSources = [];
    try {
        ...
    } catch (err) {
        if (responseText) { ... }    // ← 정상 접근
    }
}
```

### 수정 1 — `drug-info-tool.ts` AbortController timeout 3곳

| 위치 | 타임아웃 | 에러 처리 |
|------|---------|---------|

모든 수정은 기존 catch/finally 구조를 유지하며 try/finally + clearTimeout 패턴으로 일관되게 적용됨.
