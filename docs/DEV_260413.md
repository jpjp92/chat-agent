## 원인 분석

### 실행 흐름

```
[useChatStream] 이미지 전송
  └─ /api/chat → Vercel Function 시작 (60s 카운트 시작)
       │
       ├─ [router.ts] gemini-2.5-flash-lite LLM 호출 (~1s)
       │              ← SSE 무음 구간 (클라이언트에 아무것도 전송 안 됨)
       │
       ├─ [generator.ts] sdkContents 빌드
       │    └─ 공개 URL 이미지 → fetch() → base64 변환 (~2~5s)
       │                         ← SSE 무음 구간 계속
       │
       ├─ [Gemini API] generateContentStream (~10~40s)
       │    └─ 이 시점부터 SSE 청크 전송 시작
       │
       └─ [chat.ts] 스트리밍 완료 후 Supabase 2회 순차 await ← 핵심
            await chat_messages.insert(...)   ← Vercel 60s 한계 소진
            await chat_sessions.update(...)
```

### 확인된 원인

#### ① Latency — Router 필수 LLM 호출 (`router.ts`)

모든 요청에 무조건 `gemini-2.5-flash-lite` LLM 분류 호출 (~1s).
이미지 + 의약품 키워드 없는 일반 이미지 분석은 어차피 `"general"` → Router를 건너뛸 수 있음.

#### ② Latency — 공개 URL 이미지 이중 왕복 (`generator.ts`)

3MB 이상 이미지: 클라이언트 → Supabase Storage 업로드 → 공개 URL.  
Generator가 이 URL을 서버에서 다시 `fetch()` → base64 변환 (~2~5s).  
Gemini SDK는 공개 URL을 `fileData`로 직접 지원하므로 재-fetch 불필요.

#### ③ 세션 종료 — Supabase 순차 await이 60s 한계 소진 (`chat.ts`)

스트리밍 완료 후 Supabase 쓰기 2회를 순차 `await`.  
스트리밍에 50s+ 소요 시 → Vercel 60s에 함수 강제 종료 → `res.end()` 미호출 → SSE abrupt close.  
클라이언트 `reader.read()` → `done: true` 에러 없이 탈출 → `modelResponse` 없으면 AI 메시지 미추가 → **응답 소실 = "세션 종료"처럼 보임.**

#### ④ 세션 종료 — SSE heartbeat 없음

Router + image re-fetch 구간 동안 클라이언트에 SSE 이벤트가 전혀 없음.  
일부 프록시/환경에서 무음 구간에 연결을 끊을 수 있음.

## 수정 결과

### 수정 1 — Supabase 쓰기 fire-and-forget (`api/chat.ts`) ✅

스트리밍 완료 즉시 `res.end()`, Supabase 쓰기는 `.then()` 비동기 처리.  
`finally { clearInterval(heartbeatInterval) }` 블록으로 heartbeat 타이머 정리.

```typescript
// 수정 전
await supabase.from('chat_messages').insert({ ... });
await supabase.from('chat_sessions').update({ ... });
res.end();

// 수정 후
res.end();  // 스트리밍 완료 즉시 응답 종료
supabase.from('chat_messages').insert({ ... }).then(({ error }) => { ... });
supabase.from('chat_sessions').update({ ... }).then(({ error }) => { ... });
```

### 수정 2 — SSE heartbeat (`api/chat.ts`) ✅

Router/Vision 실행 중 15s마다 `{ heartbeat: true }` 이벤트 전송.  
`geminiService.ts` SSE 파서에서 `if (data.heartbeat) continue`로 무시.

```typescript
const heartbeatInterval = setInterval(() => {
    res.write(`data: ${JSON.stringify({ heartbeat: true })}\n\n`);
}, 15000);
```

### 수정 3 — 공개 URL 이미지 fileData 직접 전달 (`generator.ts`) ✅

Supabase Storage 공개 URL은 `fileData: { fileUri: url, mimeType }`로 직접 전달.  
서버 side fetch → base64 변환 (~2~5s) 제거.  
mimeType은 URL 확장자로 추론 (기본 `image/jpeg`).

```typescript
// 수정 전
const fetchRes = await fetch(url);
const b64 = Buffer.from(await fetchRes.arrayBuffer()).toString('base64');
parts.push({ inlineData: { mimeType, data: b64 } });

// 수정 후
parts.push({ fileData: { fileUri: url, mimeType: mimeTypeHint } });
```

### 수정 4 — Vision Node URL 처리 동일 수정 (`vision.ts`) ✅

drug_id intent에서 Vision Node도 공개 URL 이미지를 재-fetch하던 문제 수정.  
`image_url` 타입으로 LangChain에 URL 직접 전달.

```typescript
// 수정 전: URL → fetch → base64
if (inlineData.startsWith("http")) {
    const res = await fetch(inlineData);
    inlineData = Buffer.from(await res.arrayBuffer()).toString('base64');
}

// 수정 후: URL 직접 전달
if (imageAtt.data.startsWith("http")) {
    imageUrlPart = { type: "image_url", image_url: { url: imageAtt.data } };
}
```

### 수정 5 — Router fast-path (`router.ts`) ✅

이미지 첨부 + 의약품 키워드 없음 → `"general"` 확정, Router LLM 호출 스킵 (~1s 단축).  
이미지 첨부 + 의약품 키워드 있음 → 기존대로 LLM 분류 진행 (`drug_id` 판단).

```typescript
const hasMedicalKeyword = medicalKeywords.some(k => textContent.includes(k)) || ...;
if (hasImage && !hasMedicalKeyword) {
    console.log('[LangGraph] Router fast-path: image without medical keyword → general');
    return { nextNode: "generator", intent: "general" };
}
```
