## Fix A — `DrugRenderer.tsx` null crash 수정

**현상**: 약품 카드 렌더링 시 `Uncaught TypeError: Cannot read properties of null (reading 'split')` 발생 → 앱 전체 크래시.

**원인**: `data.category`가 null인 약품(MFDS 미등록, ConnectDI 폴백 성공 케이스 등)에서 `.split()` 직접 호출.

```tsx
// 수정 전 (null crash)
{data.category.split(/[,/()]+/).filter(Boolean).map(...)}

// 수정 후
{(data.category || '').split(/[,/()]+/).filter(Boolean).map(...)}
```

**추가 수정**: `data.ingredient` null 렌더링도 함께 처리.

```tsx
// 수정 전
{data.ingredient}

// 수정 후
{data.ingredient || '-'}
```

**효과**: ConnectDI 폴백으로 이미지를 가져왔지만 MFDS 데이터가 없는 약품(category/ingredient null)에서도 약품 카드 정상 렌더링.

## Fix B — `drug-info-tool.ts` PARTIAL_DATA 블록 제거 → 웹 검색 폴백

**기존 접근**: MFDS 결과 없음 + pharm.or.kr URL 발견 시 `PARTIAL_DATA` 지시문으로 LLM에 부분 데이터 전달 → `[MANDATORY] json:drug 반드시 생성` 강제.

**문제**: LLM이 필수 필드가 없는 상태에서 json:drug를 강제 생성하면 모든 필드가 null인 빈 약품 카드만 반환. 렌더러는 카드를 표시하지만 내용이 없어 UX상 더 나쁜 결과.

**수정**: PARTIAL_DATA 블록 전체 제거. MFDS 결과 없는 경우 `search_web` 툴 사용 지시로 단순화.

```ts
// 수정 전 (PARTIAL_DATA 강제 drug card)
if (!Array.isArray(items) || items.length === 0) {
    // pharm.or.kr URL 발견 시 PARTIAL_DATA 지시문으로 drug card 생성 시도
    const pharmUrl = await pharmUrlPromise.catch(() => null);
    if (pharmUrl) {
        return `[PARTIAL_DATA]\n...[MANDATORY] json:drug 반드시 생성...`;
    }
}

// 수정 후 (단순 웹 검색 폴백)
if (!Array.isArray(items) || items.length === 0) {
    pharmUrlPromise.catch(() => {}); // 무시
    console.log(`[Agent Tool] MFDS returned no results for "${drug_name}". Falling back to web search.`);
    return `식약처 DB에서 "${drug_name}" 정보를 찾지 못했습니다. search_web 툴로 "${drug_name} 성분 효능 용법 용량"을 검색하여 일반 텍스트로 정보를 제공하세요. json:drug 블록은 생성하지 마세요.`;
}
```

**효과**: MFDS에 없는 약품 → 일반 웹 검색 결과로 텍스트 응답. null 필드로 가득한 빈 카드 대신 실질적인 정보 제공.

## Fix C — `fetch-url.ts` URL 콘텐츠 추출 개선

**현상**: URL 요약이 지나치게 간단하게 나옴. 뉴스 기사 등 본문이 있는 페이지에서도 메타 정보 수준의 짧은 요약만 반환.

**원인**: 기존 구현은 raw HTML에서 태그만 제거하고 잘라냄. 네비게이션, 푸터, 스크립트 노이즈가 포함되어 실제 본문 비율이 낮음. og: 메타 정보도 미추출.

**수정 내용**:

### 1. 브라우저 요청 헤더 추가

```ts
headers: {
    'User-Agent': 'Mozilla/5.0 ... Chrome/120.0.0.0 Safari/537.36',
    'Accept': 'text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8',
    'Accept-Language': 'ko-KR,ko;q=0.9,en-US;q=0.8,en;q=0.7',
}
```

일부 서버의 Bot 차단 우회. 한국어 콘텐츠 우선 수신.

### 2. og: 메타 정보 추출

```ts
const ogTitle = html.match(/<meta[^>]+property=["']og:title["'][^>]+content=["']([^"']+)["']/i)?.[1]
             || html.match(/<title[^>]*>([^<]+)<\/title>/i)?.[1] || '';
const ogDesc  = html.match(/<meta[^>]+property=["']og:description["'][^>]+content=["']([^"']+)["']/i)?.[1]
             || html.match(/<meta[^>]+name=["']description["'][^>]+content=["']([^"']+)["']/i)?.[1] || '';
```

### 3. 노이즈 태그 제거

```ts
let cleaned = html
    .replace(/<script\b[^<]*(?:(?!<\/script>)<[^<]*)*<\/script>/gi, '')
    .replace(/<style\b[^<]*(?:(?!<\/style>)<[^<]*)*<\/style>/gi, '')
    .replace(/<(nav|header|footer|aside|iframe|noscript|figure|form)[^>]*>[\s\S]*?<\/\1>/gi, '');
```

### 4. 시맨틱 콘텐츠 블록 우선 추출

```ts
const articleMatch =
    cleaned.match(/<article[^>]*>([\s\S]*?)<\/article>/i) ||
    cleaned.match(/<main[^>]*>([\s\S]*?)<\/main>/i) ||
    cleaned.match(/class=["'][^"']*(?:article[-_]content|post[-_]content|news[-_]body|content[-_]area)[^"']*["'][^>]*>([\s\S]*?)<\/(?:div|section)>/i);

const bodyHtml = articleMatch ? (articleMatch[1] || articleMatch[0]) : cleaned;
```

추출 우선순위: `<article>` → `<main>` → class 패턴 → 전체 HTML.

### 5. 최종 콘텐츠 조합

```ts
let content = '';
if (ogTitle) content += `제목: ${ogTitle.trim()}\n`;
if (ogDesc) content += `요약: ${ogDesc.trim()}\n\n`;
content += bodyText;

return res.status(200).json({ content: content.trim().slice(0, 20000) });
```

**효과**: 뉴스 기사에서 og:title + og:description + 본문 전문 추출. 네비게이션/광고 노이즈 제거로 실질 콘텐츠 비율 대폭 향상. LLM 요약 품질 개선 확인.

## Fix D — `chat.ts` API 키 소진 에러 분류 추가

**배경**: 모든 API 키가 rate-limit 상태일 때 `config.ts`는 `null`을 반환하고 `generator.ts`는 `'No API key available'` 등의 메시지와 함께 throw. 기존 `chat.ts` 에러 분류에 이 케이스가 없어 `'응답 생성 중 문제가 발생했습니다.'` 일반 메시지 반환.

**수정**: API 키 소진 에러 메시지 패턴 감지 → 사용자 관점에서 과부하와 동일하게 처리.

```ts
const msg = error?.message ?? '';
const clientMessage =
  status === 429 || msg.includes('429') || msg.includes('RESOURCE_EXHAUSTED')
    ? '요청이 많아 잠시 지연되고 있습니다. 잠시 후 다시 시도해주세요.'
  : msg.includes('No API key available') || msg.includes('API keys exhausted') || msg.includes('All API keys')
    ? '요청이 많아 잠시 지연되고 있습니다. 잠시 후 다시 시도해주세요.'
  : status === 503 || msg.includes('503') || msg.includes('UNAVAILABLE')
    ? '서버가 일시적으로 불안정합니다. 잠시 후 다시 시도해주세요.'
  : status === 401 || status === 403
    ? '인증 오류가 발생했습니다. 관리자에게 문의해주세요.'
  : '응답 생성 중 문제가 발생했습니다. 다시 시도해주세요.';
```

**효과**: 키 소진 시 일관된 "잠시 지연" 메시지 표시. 내부 에러 문자열 노출 없음.

## Fix E — `drug-info-tool.ts` MFDS 미등록 약품 출처 미표시 수정

**현상**: MFDS에 없는 약품(예: 라베피드정) 조회 시 [1], [2] 인용 번호는 나오지만 출처 칩(source chip)이 표시되지 않음. LLM이 훈련 데이터 기반 가짜 인용 생성.

**원인**: `searchDrugInfoTool`이 MFDS 결과 없을 때 "search_web 툴을 호출하세요"라는 지시만 반환. LLM이 실제로 `search_web` 툴을 호출하지 않고 직접 응답을 생성해버림 → 실제 웹 검색 없이 가짜 인용 [1], [2] 사용.

**수정 파일**: `api/_lib/agent/drug-info-tool.ts`, `api/chat.ts`

**수정 내용**:

1. **`drug-info-tool.ts`**: LLM에 "search_web 써줘" 지시 대신 `searchWebTool.invoke()` 직접 호출 후 결과(+`[WEB_SOURCE_URLS]`) 임베딩해서 반환.

```ts
// 수정 전 (LLM이 무시하고 가짜 인용 생성)
return `...search_web 툴로 "${drug_name} 성분 효능 용법 용량"을 검색하여...`;

// 수정 후 (도구 내부에서 직접 검색 수행)
import { searchWebTool } from "./tools.js";
// ...
const webResult = await searchWebTool.invoke({ query: `${drug_name} 성분 효능 용법 용량` });
return `[MFDS_NOT_FOUND] 식약처 DB에서 "${drug_name}" 공식 정보를 찾지 못했습니다. 아래 웹 검색 결과를 바탕으로 일반 텍스트로 안내하세요. json:drug 블록은 생성하지 마세요.\n\n${webResult}`;
```

2. **`chat.ts`**: `on_tool_end` URL 추출 조건 확장 — `search_drug_info` 이름의 이벤트에서도 `[WEB_SOURCE_URLS]` 블록 파싱.

```ts
// 수정 전
} else if (event.event === "on_tool_end" && event.name === "search_web") {

// 수정 후
} else if (event.event === "on_tool_end" && (event.name === "search_web" || event.name === "search_drug_info")) {
```

**동작 흐름 변경**:
```
[전] searchDrugInfoTool → "search_web 써줘" → LLM 무시 → 가짜 [1][2] 생성 → 출처 칩 없음
[후] searchDrugInfoTool → DDG 직접 검색 → 실제 결과 + URL 반환 → LLM 실제 데이터로 응답 → 출처 칩 표시
```

**순환 import 없음**: `tools.ts` → `drug-info-tool.ts` 방향 없음. `drug-info-tool.ts` → `tools.ts` 단방향.

## Fix F — URL 요약 품질 개선 (Google Search 비활성화)

**현상**: URL 요약 시 응답이 지나치게 짧음. 전문이 있어도 2~3문장 수준의 요약만 반환.

**원인**: `generator.ts`에서 `[URL_CONTENT]`가 있을 때도 Google Search가 활성화됨. LLM이 fetch-url로 가져온 전문 대신 Google Search의 짧은 스니펫을 사용해 응답 생성. 또한 `prompt.ts`에 `[URL_CONTENT]` 처리 지침이 없어 LLM이 제공된 본문을 어떻게 다뤄야 할지 명시적 안내 부재.

**수정 파일**: `api/_lib/agent/nodes/generator.ts`, `api/_lib/agent/prompt.ts`

**수정 내용**:

1. **`generator.ts`**: `[URL_CONTENT]` 감지 시 `useGoogleSearch = false`. YouTube 트랜스크립트 최적화(`hasTranscript`)와 동일 패턴.

```ts
const hasUrlContent = state.webContent.includes('[URL_CONTENT]');
// ...
if (hasUrlContent) {
    useGoogleSearch = false;
    // "[LangGraph] URL content provided — Google Search disabled to use full article text"
}
```

2. **`prompt.ts`**: `[URL_CONTENT]` 처리 지침 추가.

```
- If PROVIDED_SOURCE_TEXT contains "[URL_CONTENT]", it is the FULL TEXT of a web page the user wants analyzed.
  You MUST use this as your SOLE primary source. Provide a comprehensive, well-structured summary covering
  all major points in the article. Use headings, bullet points, and bold text for clarity.
  DO NOT rely on Google Search or training knowledge for this article's content.
```

**효과**: URL 요약 시 Google Search 비활성화 → LLM이 fetch-url로 가져온 20,000자 전문 사용 → 헤딩·불릿 포인트가 포함된 구조화된 상세 요약 출력.
