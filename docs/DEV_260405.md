## 핵심 병목 분석

### 1. LCP Element render delay 3,320ms (최우선)
**원인**: `isAuthLoading = true` 동안 `<LoadingScreen>` 렌더 → 웰컴 메시지가 DOM에 없음 → auth + sessions API 왕복(~3,300ms) 이후에야 LCP 대상 요소 생성.

```
HTML(10ms) → JS → LoadingScreen 표시
    → /api/auth
    → /api/sessions (총 ~3,300ms)
    → isAuthLoading = false → 웰컴 메시지 DOM 생성 → LCP 측정
```

**목표**: LoadingScreen 제거 → 웰컴 메시지 즉시 렌더 → LCP ~200ms 기대

### 2. /api/sessions 크리티컬 패스 2,064ms
**원인**: `await loadUserSessions()` — sessions API 응답까지 화면 진입 블로킹.

### 3. 새 채팅 / 세션 전환 지연
- **새 채팅**: `createSession()` Supabase POST 응답 대기 (~500~1500ms)
- **세션 전환**: `fetchSessionMessages()` 응답 전까지 빈 화면

### 4. FontAwesome font-display (30ms)
- `index.html`에 `@font-face` override 이미 적용 중 — cdnjs 링크 자체 처리 필요

# 시각화 카드 전체화면 팝업 (Visualization Expand Modal)

## 기능 개요
시각화 카드(BioRenderer, ChemicalRenderer 등) 우하단에 expand 버튼을 추가하여
클릭 시 모달 팝업으로 해당 시각화를 전체화면 크기로 표시.

---

## 구현 방식

### 전반 구조
```
카드 우하단 expand 버튼 (fa-expand 아이콘)
    ↓ 클릭
App.tsx expandedViz state 업데이트 → { type, data }
    ↓
createPortal(document.body)로 VisualizationModal 마운트
    - backdrop blur + z-index 최상위
    - 동일 Renderer를 fullscreen 크기로 재렌더링
    - ESC 키 / X 버튼으로 닫기
```

### 상태 구조
```typescript

    type: 'bio' | 'chemical' | 'physics' | 'constellation' | 'chart' | 'diagram' | 'drug';
    data: any;

```

### 주요 변경 파일
1. `App.tsx` — `expandedViz` state + `VisualizationModal` 컴포넌트 (Portal)
2. `components/ChatMessage.tsx` — 각 Renderer 래퍼에 expand 버튼 + `onExpand` 콜백
3. 각 Renderer 컴포넌트 — `isExpanded?: boolean` prop 수용 (크기 조정용)

### expand 버튼 위치

# 세션 Race Condition 버그 수정

## 버그 증상
새로고침 후 웰컴 화면에서 메시지를 입력하면, 새 채팅이 아닌 **기존 마지막 세션에 메시지가 추가**되는 문제.

## 원인
`loadUserSessions`에서 `handleSelectSession(mappedSessions[0].id)` 자동호출 → `currentSessionId`가 기존 세션 ID로 설정된 상태에서 웰컴 화면 표시 → 사용자가 입력 시 `handleSendMessage`가 기존 세션 ID로 메시지 추가.

## 수정 내용

### 1. `loadUserSessions` — 자동 세션 선택 제거 (`App.tsx`)
```typescript
// 변경 전
setSessions(mappedSessions);
if (mappedSessions.length > 0) handleSelectSession(mappedSessions[0].id);

// 변경 후
setSessions(mappedSessions);
// 자동 선택하지 않음 — 웰컴 화면에서 첫 메시지 전송 시 새 세션 생성
```

### 2. `handleSendMessage` — null 세션 시 새 세션 자동 생성 (`App.tsx`)
```typescript
// 변경 전
if (!currentSessionId || (!content.trim() && ...)) return;

// 변경 후
if (!content.trim() && attachments.length === 0) return;

let activeSessionId = currentSessionId;
if (!activeSessionId) {
  const { session, error } = await createSession(targetUserId);
  // 신규 세션 생성 후 setSessions + setCurrentSessionId
  activeSessionId = session.id;
}
```

### 3. 함수 내부 `currentSessionId` → `activeSessionId` 교체
`handleSendMessage` 내 `currentSessionId` 참조 9개를 `activeSessionId`로 교체 — 새로 생성된 세션 ID 기준으로 모든 state 업데이트 수행.

## 결과
- 새로고침 → 웰컴 화면 → 입력 → **새 세션 자동 생성** 후 대화 시작 ✅
- 기존 세션 선택 후 대화 시 동작 변경 없음 ✅

# 스트리밍 실시간 전송 버그 수정 (2026-04-05)

## 버그 증상
질문 전송 시 응답이 스트리밍되지 않고 **완성된 전체 텍스트가 한 번에 나타나는** 문제. 첫 전송 시 웰컴 화면으로 잠깐 되돌아가는 현상도 동반.

## 원인
SDK path(`medical_qa` / `biology` 등)의 `generator.ts`에서 `genai.models.generateContent()` (비스트리밍) 사용 중 — 응답 완성 후 단일 이벤트로 전송.

## 수정 내용

### `api/_lib/agent/nodes/generator.ts`
```typescript
// 변경 전
const result = await genai.models.generateContent({ ... });
responseText = result.text ?? '';

// 변경 후
const sdkStream = await genai.models.generateContentStream({ ... });
for await (const chunk of sdkStream) {
    const chunkText = chunk.text ?? "";
    if (chunkText) {
        responseText += sanitized;
        if (sendEvent) sendEvent({ text: sanitized }); // 청크 즉시 전송
    }
}
```
- `createGeneratorNode(systemInstructionBase, isYoutubeRequest, sendEvent?)` 파라미터 추가
- 마지막 청크에서 `groundingMetadata` 수집 후 `sendEvent({ groundingSources })` 전송

### `api/_lib/agent/graph.ts`
- `compileAgentGraph(systemInstruction, isYoutubeRequest, sendEvent?)` 파라미터 추가
- `createGeneratorNode`에 `sendEvent` 전달

### `api/chat.ts` — `trackingEvent` 패턴
```typescript
let fullAiResponse = '';
const trackingEvent = (data: any) => {
    if (data.text) fullAiResponse += data.text; // Supabase 저장용 자동 누적
    sendEvent(data);
};
const graph = compileAgentGraph(systemInstruction, isYoutubeRequest, trackingEvent);
```
- `on_chain_end(generator)`: fallback 전용 — SDK/LangChain 스트리밍으로 이미 전송된 경우 중복 방지

## 결과
- 응답이 청크 단위로 실시간 타이핑 효과로 표시됨 ✅
- LangChain path(`drug_id` / `drug_info`)는 기존 `on_chat_model_stream` 유지 ✅

# 웰컴화면 질의 미표시 버그 수정 (2026-04-05)

## 버그 증상
새 세션 생성 후 **사용자 질의 버블이 화면에 나타나지 않음**. 사이드바에서 해당 채팅을 다시 클릭해야 히스토리가 보임.

## 원인
`handleSendMessage`에서 두 번의 분리된 `setSessions` 호출 → React 배치 최적화에서 첫 번째 호출(userMessage 포함)이 두 번째 호출(빈 messages)에 덮어씌워짐.

## 수정 내용 (`App.tsx`)
```typescript
// 1. userMessage 먼저 생성
const userMessage: Message = { id, role: Role.USER, content, ... };

// 2. 새 세션 생성 경로 — userMessage를 초기 messages에 직접 포함
const newSess: ChatSession = {
    id: session.id,
    title: session.title,
    messages: [userMessage], // ← 직접 포함
    ...
};
setSessions(prev => [newSess, ...prev]); // 단일 setSessions
```

## 결과
- 새 세션 생성 즉시 사용자 질의 버블이 화면에 표시됨 ✅
- AI 응답 스트리밍도 동일 세션에 연속 표시됨 ✅

# 사이드바 ⋯ 드롭다운 메뉴 (2026-04-05)

## 문제
긴 제목을 가진 세션 항목에서 편집/삭제 버튼(`absolute right-2`)이 `truncate ...` 생략 기호 위에 겹쳐 UX 저해.

## 해결 방안 (방안 B 적용)
기존 편집/삭제 버튼 2개 → `⋯` 단일 아이콘 버튼 + 드롭다운 메뉴 방식으로 교체.

## 구현 (`components/ChatSidebar.tsx`)

### 추가 state / ref
```typescript
const [openMenuId, setOpenMenuId] = useState<string | null>(null);
const menuRef = useRef<HTMLDivElement>(null);

useEffect(() => {
    const handleClickOutside = (e: MouseEvent) => {
        if (menuRef.current && !menuRef.current.contains(e.target as Node))
            setOpenMenuId(null);
    };
    if (openMenuId) document.addEventListener('mousedown', handleClickOutside);
    return () => document.removeEventListener('mousedown', handleClickOutside);
}, [openMenuId]);
```

### 세션 아이템 구조
- 제목 `text-[13px] tracking-tight pr-5` — ⋯ 버튼 공간 확보
- `overflow-hidden` 제거 — 드롭다운이 아이템 밖으로 표시되도록
- hover 시 ⋯ 버튼 노출 → 클릭 시 `openMenuId` 토글
- 드롭다운: `w-24 rounded-xl shadow-lg`, 편집(fa-pen) + 삭제(fa-trash-can, text-red-500)

## 우선순위 1. `App.tsx` 상태 / 오케스트레이션 분리

### 현재 문제
`App.tsx`가 아래 역할을 동시에 수행 중:
- auth 초기화
- 세션 목록 / 세션 선택 / 세션 생성 / 삭제 / 제목 변경

- 스트리밍 응답 누적
- 토스트 / 다이얼로그 / 프로필 상태
- 메인 레이아웃 렌더

이 구조는 최근 발생한 다음 이슈와 직접 연결됨:
- 세션 race condition

- 자동 익명 로그인
- `currentUser`, `isAuthLoading`, auth error 상태 관리

반환 예시:
```typescript
const {

    isAuthLoading,
    authError,
    refreshAuth,
```

- `sessions`, `currentSessionId`

반환 예시:
```typescript
const {
    sessions,
    currentSession,
    currentSessionId,
    createSession,
    selectSession,
    deleteSession,
    renameSession,
```

- `handleSendMessage`

## 우선순위 3. 서비스 계층 에러 계약 통일

### 현재 문제
`services/geminiService.ts`의 fetch 래퍼들이
- 어떤 함수는 `{ data, error }` 패턴
- 어떤 함수는 `throw`
- 어떤 함수는 `response.ok` 검사 없이 `response.json()`
형태로 혼재.

상위 코드에서 `if (error)`와 `try/catch`가 섞여 사용되어 흐름 추적이 어려움.

### 해결 방안

#### 방안 A. 전부 throw 기반으로 통일
```typescript
const data = await apiFetch<T>(url, init);
```

#### 방안 B. 전부 Result 기반으로 통일
```typescript
const { data, error } = await apiFetch<T>(url, init);
```

### 권장안
현재 코드베이스는 이미 `if (error)` 패턴을 많이 쓰므로 **Result 기반 유지 + 내부 fetch wrapper 도입**이 전환 비용이 낮음.

예시:
```typescript
type ApiResult<T> = { data: T | null; error: string | null };
```

### 예상 효과
- 에러 처리 흐름 일관화
- silent failure 감소
- 신규 API 추가 시 구현 편차 감소
