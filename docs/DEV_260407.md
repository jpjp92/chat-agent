## 수정 1 — DrugRenderer race condition + sync-drug-image 스코프 버그

### 원인 A: Race condition (`components/DrugRenderer.tsx`)

`useEffect`에 cleanup이 없어, Drug A → Drug B로 전환 시 Drug A의 `syncImage()` 요청이 계속 실행됨.
Drug A 요청 실패 → `setImageError(true)` → Drug B 카드에 "이미지 준비 중" 표시.

```
Drug A 렌더 → syncImage() 시작
Drug B 렌더 → syncImage() 시작 (Drug A 요청 아직 진행 중)
  Drug A 요청 실패 → setImageError(true) ← Drug B 카드 상태 오염
  Drug B 요청 성공 → setSyncedUrl(url)   ← imageError=true라 이미 무시됨
```

### 원인 B: 스코프 버그 (`api/sync-drug-image.ts`)

`fileName`, `resolveInflight`가 `try` 블록 안에서 `const`/`let`으로 선언되어 `catch`에서 접근 불가.
예외 발생 시 inflight Promise가 `resolve`되지 않아 동일 drug의 후속 요청이 영원히 대기.

### 수정

**`components/DrugRenderer.tsx`**
- `AbortController` 추가, cleanup에서 `abort()` 호출
- `AbortError`는 state 변경 없이 무시

```typescript
const controller = new AbortController();
const syncImage = async () => {
    try {
        const response = await fetch('/api/sync-drug-image', {
            ...
            signal: controller.signal
        });
        ...
    } catch (error: any) {
        if (error.name === 'AbortError') return;
        setImageError(true);
    }
};
syncImage();
return () => controller.abort();
```

**`api/sync-drug-image.ts`**
- `fileName`, `resolveInflight` 선언을 `try` 바깥으로 이동

```typescript
let fileName: string | undefined;
let resolveInflight: ((v: { publicUrl: string; pillVisual: any } | null) => void) | undefined;

try {
    ...
    fileName = `drug-cache/${urlHash}.jpg`;
    inflightRequests.set(fileName, new Promise(r => { resolveInflight = r; }));
    ...
} catch (error: any) {
    resolveInflight?.(null);           // catch에서 정상 접근 가능
    inflightRequests.delete(fileName);
    ...
}
```

## 수정 2 — sync 실패 시 proxy fallback 무시 버그

### 원인

위 수정 1 이후에도 이미지가 표시 안 되는 경우가 남아 있음.

`sync-drug-image` 응답이 non-OK (nedrug 서버 차단, 타임아웃 등)이면 `setImageError(true)` 호출.
그런데 MFDS `itemImageDownload` URL은 `isSearchOrEntryPage = false`라서 `proxiedImageUrl`이 이미 유효한 proxy URL로 존재함.
렌더 조건 `!proxiedImageUrl || imageError` → `false || true` → "이미지 준비 중" 표시.

```
이미지 URL: https://nedrug.mfds.go.kr/pbp/.../itemImageDownload/...
isSearchOrEntryPage = false
proxiedImageUrl = "/api/proxy-image?url=..."  ← 유효
imageError = true  ← sync 실패로 설정됨

렌더 조건: !proxiedImageUrl || imageError
         = false || true
         = true  → "이미지 준비 중" ← BUG
```

**sync-drug-image 역할**: 이미지를 Supabase에 캐싱 (성능 최적화)
**proxy-image 역할**: 요청 시 실시간 프록시 (fallback)

두 API는 독립적으로 동작하므로, sync 실패 = "캐싱 안 됨"일 뿐 이미지 표시 불가가 아님.

### 수정 (`components/DrugRenderer.tsx`)

sync 실패 시 `setImageError`를 호출하지 않음. `imageError`는 `<img onError>` 핸들러에서만 설정.

```typescript
// 수정 전
} else {
    setImageError(true);  // ← sync 실패 시 imageError 설정
}
} catch (error: any) {
    if (error.name === 'AbortError') return;
    setImageError(true);  // ← 네트워크 오류 시도 imageError 설정
}

// 수정 후
// sync 실패 / 네트워크 오류 → 아무것도 하지 않음
// proxiedImageUrl(proxy)가 있으면 그대로 렌더링
// <img onError>에서만 setImageError(true) 호출
```

| 상황 | 수정 전 | 수정 후 |
|------|---------|---------|
| sync 실패 (nedrug 차단) | `imageError=true` → "이미지 준비 중" | proxy URL로 이미지 렌더 |
| proxy도 실패 (`<img onError>`) | `imageError=true` → "이미지 준비 중" | `imageError=true` → "이미지 준비 중" (정상) |
| sync 성공 | `syncedUrl` 사용 | `syncedUrl` 사용 (동일) |

## 수정 3 — `mapDbMessage` attachment 복원 불완전

> 발견: [DEV_260406.md](DEV_260406.md) 추가 발견 사항

### 원인

`api/chat.ts`에서 DB 저장 시 `attachment_url`에 두 가지 값을 저장:
- 업로드된 경우: `https://.../chat-imgs/hash.jpg` (HTTP URL)
- base64 직접 전송: `image/jpeg` (mimeType 문자열)

`mapDbMessage`는 이 구분 없이 `mimeType: attachment_url`로 사용하고 `data: ''`로 비워두어,
URL 케이스에서는 mimeType에 URL이 들어가 렌더링 판단 오류, `data`가 비어 이미지 표시 불가.

```typescript
// 수정 전
attachment: message.attachment_url ? {
    fileName: message.attachment_url.includes('pdf') ? 'document.pdf' : 'image_attached',
    mimeType: message.attachment_url,  // ← URL을 mimeType으로 사용
    data: '',                          // ← 비어 있어 렌더링 불가
} : undefined,
```

### 수정 (`src/hooks/useChatSessions.ts`)

`inferAttachment()` 헬퍼 추가:
- HTTP URL → Supabase 버킷 경로(`/chat-imgs/`, `/chat-docs/`, `/chat-videos/`)로 mimeType 추론, `data = URL` 복원
- mimeType 문자열 → 그대로 사용, `data = ''` (base64는 DB에 저장되지 않으므로 복원 불가)

```typescript
const inferAttachment = (attachmentUrl: string) => {
    if (!attachmentUrl.startsWith('http')) {
        // mimeType 문자열로 저장된 경우 (base64 직접 전송)
        const fileName = attachmentUrl.includes('pdf') ? 'document.pdf'
            : attachmentUrl.startsWith('video/') ? 'video.mp4' : 'image_attached';
        return { fileName, mimeType: attachmentUrl, data: '' };
    }
    // HTTP URL로 저장된 경우 — 버킷 경로로 mimeType 추론
    const mimeType = attachmentUrl.includes('/chat-docs/') || attachmentUrl.includes('.pdf')
        ? 'application/pdf'
        : attachmentUrl.includes('/chat-videos/') || attachmentUrl.includes('.mp4')
        ? 'video/mp4' : 'image/jpeg';
    const fileName = mimeType === 'application/pdf' ? 'document.pdf'
        : mimeType.startsWith('video/') ? 'video.mp4' : 'image_attached';
    return { fileName, mimeType, data: attachmentUrl };
};
```

## 수정 4 — `useChatStream` URL 처리 블록 로딩 스피너 영구 고착

> 발견: [DEV_260406.md](DEV_260406.md) 추가 발견 사항

### 원인

`fetchUrlContent` / `fetchYoutubeTranscript` 호출이 `streamChatResponse` `try-catch` 바깥에 위치.
이 함수들이 throw하면 `finally` 없이 상위로 전파되어 `setIsTyping(false)` / `setLoadingStatus(null)`이
호출되지 않아 로딩 스피너 영구 고착.

```
URL 처리 블록 (try-catch 없음)
  ↓ fetchUrlContent() ← throw 발생
  ↓ (finally 없음)
  ↓ 함수 탈출
     setIsTyping(false) 미호출 → 스피너 고착
     setLoadingStatus(null) 미호출 → 로딩 텍스트 고착
```

### 수정 (`src/hooks/useChatStream.ts`)

YouTube / 일반 URL 처리 블록을 각각 독립 `try-catch`로 감쌈:

```typescript
} else if (isYoutube) {

}

if (urlFetchError) {
    onError('URL 콘텐츠를 가져오지 못했습니다.');  // Toast 표시

```

URL fetch 실패 시에도 스트리밍은 URL 컨텍스트 없이 계속 진행 (사용자 입력 최대한 처리).

## 수정 5 — MFDS에 없는 의약품 이미지 미표시 (노즈쏙정 등)

### 로그

```
[Agent Tool] MFDS Strategy 1 (Spaceless): 노즈쏙정  → 결과 없음
[Pharm.or.kr] Best match found (Score 100): idx=33923 → 발견했지만 LLM에 미전달
→ searchWebTool 2회 fallback → image_url = null → "이미지 준비 중"
```

### 원인

MFDS API에 없는 의약품(일부 OTC)은 Pharm.or.kr에만 등록된 경우 존재.
기존 코드는 MFDS 실패 시 `"찾지 못했습니다"` 메시지를 반환하며, Pharm.or.kr에서 찾은 URL을 버렸음.

```
MFDS 실패 → pharmUrlPromise(Pharm.or.kr) 결과를 기다리지 않고 즉시 반환
           → pharmUrl이 있어도 LLM에 전달 안 됨
           → LLM: searchWebTool fallback → image_url = null
```

### 수정 1 — `api/_lib/agent/drug-info-tool.ts`

MFDS 실패 시 `pharmUrlPromise`를 `await`해서 결과가 있으면 `PARTIAL_DATA`로 LLM에 전달:

```typescript
// 수정 전
if (!Array.isArray(items) || items.length === 0) {
    return `식약처 DB에서 "${drug_name}"에 해당하는...`;
}

// 수정 후
if (!Array.isArray(items) || items.length === 0) {
    const fallbackPharmUrl = await pharmUrlPromise;

    if (fallbackPharmUrl) {
        return `...
[PARTIAL_DATA] Pharm.or.kr에서 약품 항목 발견:
Pharm_URL: ${fallbackPharmUrl}
ConnectDI_URL: ${connectdiUrl}

json:drug 생성 시 아래 필드를 반드시 포함하세요:
- "pharm_url": "${fallbackPharmUrl}"
- "image_url": "${fallbackPharmUrl}"  ← sync가 자동으로 이미지 추출
- "connectdi_url": "${connectdiUrl}"`;
    }

    return `식약처 DB에서 "${drug_name}"에 해당하는...`;
}
```

`image_url`에 pharm.or.kr 페이지 URL을 넣으면 `isSearchOrEntryPage = true`가 되어 `sync-drug-image`가 스크래핑을 시도.

### 수정 2 — `api/sync-drug-image.ts`

pharm.or.kr 페이지 스크래핑 추가 (`isPharmOrKr` 분기):

```typescript
const isPharmOrKr = finalUrl.includes('pharm.or.kr');

if (contentType.includes('text/html') && (isNaverEntry || isConnectDIEntry || isPharmOrKr)) {
    if (isPharmOrKr) {
        // health.kr CDN 이미지 추출: _b.jpg(큰 이미지) 우선, 없으면 _s.jpg
        const healthKrBig = html.match(/https?:\/\/common\.health\.kr\/shared\/images\/sb_photo\/[^"'\s>]+_b\.jpg/i);
        targetScrapedUrl = healthKrBig?.[0]
            ?? html.match(/https?:\/\/common\.health\.kr\/shared\/images\/sb_photo\/[^"'\s>]+\.jpg/i)?.[0]
               ?.replace(/_s\.jpg$/i, '_b.jpg');  // _s → _b 프로모션

        // _b.jpg 404 시 _s.jpg로 재시도
        if (targetScrapedUrl && !(await fetch(targetScrapedUrl, ...)).ok) {
            targetScrapedUrl = targetScrapedUrl.replace(/_b\.jpg$/i, '_s.jpg');
        }
    }
}
```

| 약품 유형 | image_url | sync 처리 |
|-----------|-----------|-----------|
| MFDS 있음 | `nedrug.mfds.go.kr/...itemImageDownload/...` | 직접 다운로드 |
| MFDS 없음, Pharm.or.kr 있음 | `pharm.or.kr/...show.asp?idx=...` | HTML 스크래핑 → health.kr CDN 이미지 |
| 둘 다 없음 | `null` | "이미지 준비 중" |
