# 코드 보안 및 취약점 검토 보고서 (2026-04-27)

최근 3일간(4/24~4/26)의 개발 로그와 현재 코드를 종합 검토한 결과, 개선된 사항과 여전히 남아있는 보안 취약점을 다음과 같이 정리합니다.

## 1. 개선 완료 및 조치된 사항
*   **SSRF 차단 (`api/fetch-url.ts`, `api/proxy-image.ts`):** `localhost`, `169.254.x.x` 등 사설 IP 및 메타데이터 엔드포인트에 대한 hostname 기반 차단 로직이 적용되었습니다. (4/25)
*   **보안 헤더 설정 (`vercel.json`):** `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, `HSTS` 등 주요 보안 헤더가 반영되었습니다. (4/25)
*   **의존성 업데이트:** `npm audit fix`를 통해 `undici`, `tar`, `vite` 등 주요 패키지의 취약점 일부가 해소되었습니다. (4/25)
*   **출력 제한 상향:** `maxOutputTokens`를 8,192에서 32,768로 상향하여 응답 잘림 현상을 완화했습니다. (4/25)
*   **인라인 인용 마커 제거:** 스트리밍 청크 경계에서 잘리는 인용 마커(`[N]`)를 버퍼링을 통해 완벽히 제거하도록 로직이 개선되었습니다. (4/26)

## 2. 여전히 남아있는 고위험 취약점 (CRITICAL / HIGH)
*   **IDOR (Insecure Direct Object Reference) (`api/sessions.ts`):** 
    *   `session_id`와 `user_id`를 단순히 쿼리 파라미터로 받아 인증 없이 DB를 조회/삭제합니다.
    *   타인의 `session_id`를 추측하여 대화 내용을 열람하거나 삭제할 수 있는 치명적인 위험이 있습니다.
*   **닉네임 기반 인증의 한계 (`api/auth.ts`):**
    *   단순 닉네임만으로 사용자를 생성하고 식별하므로, 중복 체크 외에 실제 소유자 확인 수단이 없습니다. (사칭 위험)
*   **관리자 권한(Service Role) 오남용:**
    *   `api/upload.ts` 등에서 여전히 `supabaseAdmin`을 폴백으로 사용하거나, RLS를 우회하는 로직이 존재합니다.
*   **보안 수정 불가 패키지 (`xlsx`):**
    *   `xlsx` 패키지는 Prototype Pollution 취약점이 있으나 현재 fix가 없습니다. 보안이 강화된 대안 패키지(예: `exceljs`)로의 교체가 시급합니다.

## 3. 추가 개선 필요 사항 (MEDIUM / LOW)
*   **입력 길이 제한 부재:** 닉네임 길이나 대화 프롬프트 길이에 대한 서버 측 제한이 없어 리소스 고갈 공격에 취약할 수 있습니다.
*   **버킷(Bucket) 화이트리스트 미비 (`api/upload.ts`):** 업로드 시 `bucket` 파라미터를 검증 없이 사용하고 있어, 의도치 않은 버킷에 대한 접근 가능성이 있습니다.
*   **Rate Limiting 부재:** 전반적인 API 엔드포인트에 대한 속도 제한이 없어 자동화된 공격(Brute-force, DoS)에 노출되어 있습니다.
*   **CSP (Content Security Policy) 미적용:** 외부 리소스(KaTeX, Fonts 등) 의존성으로 인해 아직 적용되지 못했습니다.

## 향후 우선 조치 과제
1.  **세션 보안 강화:** `api/sessions.ts`에서 요청자와 세션 소유자의 일치 여부를 검증하는 로직 추가 (IDOR 방어)
2.  **`xlsx` 패키지 교체:** 보안 취약점이 없는 라이브러리로 대체
3.  **입력값 검증 강화:** 닉네임/프롬프트 길이 제한 및 버킷 명칭 화이트리스트 도입
4.  **인증 시스템 고도화:** 장기적으로 닉네임 외에 추가적인 인증 수단 도입 검토

# 약품 카드 "자세히" 버튼 복구 + 모바일 응답 안정화 (2026-04-27)

## 문제 1 — "자세히" 버튼 사라짐

### 원인 분석

- v4.52(J5)에서 pharm.or.kr 23개 약품 전수 테스트 0% 성공 확인 후 dead code 전면 제거
- `drug-info-tool.ts` instruction 12: `"pharm_url": always null` 강제
- `DrugRenderer.tsx` 버튼 조건 `{data.pharm_url && ...}` → pharm_url이 항상 null → 버튼 미렌더
- ConnectDI URL을 임시 연결했으나 ConnectDI는 **검색 목록 페이지** + 원래 **하단 소스 칩** 위치

### 해결

pharm.or.kr 대신 **MFDS 식약처 nedrug 공식 상세 페이지** 직링크로 교체.

```
https://nedrug.mfds.go.kr/pbp/CCBBB01/getItemDetail?itemSeq=ITEM_SEQ
```

- MFDS API 응답의 `ITEM_SEQ`(품목일련번호) 활용 — 항상 반환, 서버 스크래핑 불필요
- pharm.or.kr의 idx 서버사이드 POST 스크래핑 방식 대비 훨씬 안정적

### 변경 파일

#### `api/_lib/agent/drug-info-tool.ts`
- `ITEM_SEQ` 및 `MFDS_DETAIL_URL` LLM 출력에 추가
- MANDATORY INSTRUCTIONS:
  - instruction 13: `"mfds_url"` ← MFDS_DETAIL_URL ("자세히" 버튼용)
  - instruction 14: `"connectdi_url"` ← ConnectDI_URL (소스 칩용, 기존 역할 유지)
  - instruction 15: 후보 선택 기준 (번호 재정렬)

#### `api/_lib/agent/prompt.ts`
- json:drug 스키마에 `mfds_url` 필드 추가 (pharm_url 자리 대체)
- MFDS_URL 규칙 신규 추가: MFDS_DETAIL_URL 사용, "자세히" 버튼 구동
- CONNECTDI_URL 규칙: 소스 칩 참조용으로 명시

#### `components/DrugRenderer.tsx`
- `DrugData` 인터페이스에 `mfds_url?: string`, `connectdi_url?: string` 추가
- "자세히" 버튼: `data.pharm_url` → `data.mfds_url || data.pharm_url`
- 버튼 텍스트 `t.details` i18n 적용

### 최종 링크 구조

| 요소 | 링크 | 위치 |
|------|------|------|
| "자세히" 버튼 | nedrug 식약처 공식 상세 페이지 | 카드 우측 하단 |
| ConnectDI 칩 | connectdi.com 검색 목록 | 카드 하단 소스 칩 (기존 유지) |

## 문제 2 — 모바일 응답 실패 (간헐적)

### 원인 분석

약품 인텐트 LangChain 경로 시간 누적이 Vercel 60s에 근접:

```
Router LLM 호출         ~1–2s
MFDS API (최대 8s)      ~3–8s
Vision 각인 추출 (선택)  ~3–7s
LLM 최종 생성 (32768)    ~15–30s?
─────────────────────────────
합계                     최대 ~47s / Vercel 60s
```

v4.54에서 `maxOutputTokens 32768` 상향 시 LangChain 경로(약품)도 동일 적용됐으나, 약품 카드 JSON + 한 줄 요약은 1,500토큰 이내로 충분 → 불필요하게 긴 생성 대기

### 변경 파일

#### `api/_lib/agent/nodes/generator.ts`
- LangChain 경로(drug_id / drug_info) `maxOutputTokens: 32768 → 8192`
- SDK 경로(일반 쿼리) 32768 유지
