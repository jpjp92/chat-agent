## 3. 세션 초기화 "연결에 실패했습니다" 에러 화면 수정

### 원인 분석

**경로 1 — `initAuth().catch()` 핸들러** (`useAuthSession.ts`)

`initAuth()` 내부에서 예외 발생 시 catch 블록이 `setIsAuthLoading(false)`만 호출하고 `setCurrentUser`를 명시 호출하지 않아, `isAuthLoading=false + currentUser=null` 조합으로 에러 화면 표시.

**경로 2 — localStorage 파싱 실패 후 `else` 미진입** (`useAuthSession.ts`)

`JSON.parse(savedUser)` 실패 시 `catch` 블록에서 localStorage만 제거하고 신규 게스트 생성(`else` 분기)으로 진입하지 않아 `user=null`인 채로 진행.

**경로 3 — `handleReset` React 상태 선행 업데이트** (`App.tsx`) ← 실제 재현 경로

```ts
// Before: clearStoredUser()가 setCurrentUser(null) 호출 → reload 전에 리렌더 발생
const handleReset = () => {
    clearStoredUser();           // → setCurrentUser(null), isAuthLoading=false 그대로
    window.location.reload();    // 리렌더 후 실행
};
// 결과: isAuthLoading=false + currentUser=null → 에러 화면 0.1초 순간 표시
```

### 수정 내용

**`App.tsx` — `handleReset` React 상태 업데이트 제거**

```ts
// After: React 상태 변경 없이 localStorage만 직접 제거 후 reload

- `.catch()` 핸들러에 `setCurrentUser(null)` 명시 추가

    .replace(/\[\s*,/g, '[')
    .replace(/,\s*\]/g, ']');
const jsonData = JSON.parse(repaired);

// catch: raw JSON 노출 대신 조용히 skip
} catch (e) {
    // 유효하지 않은 viz 블록은 무시 — 텍스트 설명은 그대로 표시됨

```

## 10. BioRenderer React StrictMode double-invoke 수정 (`components/BioRenderer.tsx`)

> 2026-05-03

### 문제

개발 환경(React StrictMode)에서 3D 구조가 캔버스에 표시되지 않던 문제.

**근본 원인**: React StrictMode의 `doubleInvokeEffectsOnFiber`가 effect를 두 번 실행함.

1. **Mount 1** — `loadFile('rcsb://…')` 비동기 시작 + NGL Stage 생성
2. **StrictMode cleanup** — `[]` effect cleanup이 `nglStage.current.dispose()` 호출 → WebGL 컨텍스트 파괴
3. **Mount 2** — 새 Stage 생성 후 `loadFile` 재시작 → 성공 시 `onLoaded` 호출
4. **문제**: Mount 2의 `onLoaded` 내부에서 `stageRef.current`가 null이거나 NGL Stage가 dispose 후 재생성된 직후라 WebGL 렌더링이 정상 작동하지 않음

콘솔에서 확인된 증거:
```
[NGL] stageRef: undefined x undefined   ← stageRef.current가 null
STAGE LOG loaded '1MBN.cif'             ← 파일은 로드됐지만 캔버스 blank
```

### 수정 1: `cancelled` 플래그 (Mount 1 콜백 차단)

Mount 1의 in-flight 비동기 콜백이 Mount 2의 stageRef를 침범하지 못하도록 차단.

```ts
useEffect(() => {
    if (type !== 'pdb' || !data.pdbId || !stageRef.current) return;

    let cancelled = false;

    const onLoaded = (comp: any) => {
        if (cancelled || !stageRef.current) return;  // ← Mount 1 차단
        comp.addRepresentation("cartoon", { colorScheme: "residueindex", quality: "high" });
        setTimeout(() => {
            if (cancelled || !stageRef.current) return;
            stage.handleResize();
            comp.autoView(0);
            if (!cancelled) setIsLoading(false);
            requestAnimationFrame(() => {
                if (cancelled) return;
                stage.handleResize();
                (stage as any).viewer?.render?.(false);
                (stage as any).viewer?.requestRender?.();
            });
        }, 600);
    };

    stage.loadFile(`rcsb://${data.pdbId}`)
        .then(onLoaded)
        .catch(() => {
            if (cancelled) return Promise.resolve(null);  // ← Mount 1 차단
            return stage.loadFile(`https://files.rcsb.org/download/${data.pdbId}.cif`, { ext: 'cif' });
        })
        .then((comp: any) => { if (comp && !cancelled) onLoaded(comp); })
        .catch((err) => {
            if (cancelled) return;
            console.error("[BioRenderer] Both RCSB endpoints failed:", err);
            setError("RCSB 서버가 응답하지 않습니다. 잠시 후 다시 시도해주세요.");
            setIsLoading(false);
        });

    return () => {
        cancelled = true;  // ← cleanup 시 모든 콜백 차단
        // ... listeners removed
    };
}, [type, data.pdbId]);
```

### 수정 2: `setTimeout(0)` 패턴으로 dispose 보호 (핵심 수정)

StrictMode double-invoke 시 `[]` cleanup이 `nglStage.dispose()`를 호출해 WebGL 컨텍스트가 파괴되는 문제 해결.

**원리**: StrictMode double-invoke는 cleanup → re-run을 동기적으로 실행. `setTimeout(0)` dispose는 re-run이 즉시 `clearTimeout`으로 취소함 → Stage 생존. 실제 unmount 시엔 re-run이 없으므로 timeout이 발동해 정상 dispose.

```ts
const disposeTimer = useRef<ReturnType<typeof setTimeout> | null>(null);

// 기존 [] cleanup effect 대체
useEffect(() => {
    if (disposeTimer.current !== null) {
        clearTimeout(disposeTimer.current);
        disposeTimer.current = null;
    }
    return () => {
        disposeTimer.current = setTimeout(() => {
            if (nglStage.current) {
                nglStage.current.dispose();
                nglStage.current = null;
            }
            disposeTimer.current = null;
        }, 0);
    };
}, []);
```

| 시나리오 | 동작 |
|----------|------|
| StrictMode double-invoke | cleanup → setTimeout 예약 → re-run이 `clearTimeout` → dispose 취소 → Stage 생존 |
| 실제 unmount | cleanup → setTimeout 예약 → 컴포넌트 사라짐 → timeout 발동 → dispose ✓ |

### 수정 3: render 순서 변경 (backdrop-blur 간섭 방지)

loading spinner의 `backdrop-blur-md`가 WebGL 렌더링을 방해하는 현상 방지.

```ts
// Before: requestRender() → rAF×2 → setIsLoading(false)
// After:  setIsLoading(false) → rAF → render(false) + requestRender()
if (!cancelled) setIsLoading(false);
requestAnimationFrame(() => {
    if (cancelled) return;
    stage.handleResize();
    (stage as any).viewer?.render?.(false);   // 동기 render
    (stage as any).viewer?.requestRender?.(); // 큐 render (fallback)
});
```

spinner 제거 후 깨끗한 컨텍스트에서 NGL render 실행.

## 12. YouTube Shorts URL 매칭 수정 (`api/chat.ts`)

> 2026-05-03

### 문제

YouTube Shorts URL(`youtube.com/shorts/VIDEO_ID`)이 `ytRegex`에 매칭되지 않아 Gemini native video 분석이 비활성화되는 버그.

**흐름:**
1. `fetch-url.ts`: `url.includes('youtube.com')` 로 Shorts 감지 → `[YOUTUBE_VIDEO_INFO]` 반환 (제목·채널·설명만)
2. `chat.ts` `ytRegex`: Shorts URL 미매칭 → `isYoutubeFromPrompt = false` → **`fileData` 미전송**
3. Gemini: `fileData` 없음 → 프롬프트 fallback 조건 충족 → "현재 자막 데이터를 직접 추출할 수 없어 영상의 메타데이터와 검색 결과를 바탕으로 요약을 구성했습니다" 간헐적 출력

**기존 regex가 Shorts를 못 잡는 이유:**
```
[^\/]+\/.+\/  → trailing slash 필요, shorts/VIDEO_ID 에는 없음
(?:v|embed)\/  → 'shorts' != 'v' 또는 'embed'
.*[?&]v=       → ?si=... 에는 v= 파라미터 없음
```

### 수정

```ts
// Before
const ytRegex = /(?:https?:\/\/)?(?:www\.)?(?:youtube\.com\/(?:[^\/]+\/.+\/|(?:v|e(?:mbed)?)\/|.*[?&]v=)|youtu\.be\/)([a-zA-Z0-9_-]{11})/;

// After — shorts\/ 추가
const ytRegex = /(?:https?:\/\/)?(?:www\.)?(?:youtube\.com\/(?:shorts\/|[^\/]+\/.+\/|(?:v|e(?:mbed)?)\/|.*[?&]v=)|youtu\.be\/)([a-zA-Z0-9_-]{11})/;
```

매칭 테스트:

| URL | 수정 전 | 수정 후 |
|-----|---------|---------|
| `youtube.com/shorts/VIDEO_ID` | ✗ | ✓ |
| `youtube.com/shorts/VIDEO_ID?si=...` | ✗ | ✓ |
| `www.youtube.com/shorts/VIDEO_ID` | ✗ | ✓ |
| `youtube.com/watch?v=VIDEO_ID` | ✓ | ✓ |
| `youtu.be/VIDEO_ID` | ✓ | ✓ |

### 수정 후 동작

- Shorts URL → video ID 추출 → `watch?v=VIDEO_ID`로 변환 → `fileData`로 Gemini 전송
- Gemini가 영상을 직접 시청·분석 → 실제 내용 기반 요약 + 타임스탬프 생성
- `isYoutubeRequest && hasVideoData` 조건 충족 → `thinkingBudget: 0` 적용 (Vercel 60s 타임아웃 방지)
- Shorts는 60초 이하라 타임스탬프가 `[00:00]~[00:59]` 구간에 분포하는 것이 정상
