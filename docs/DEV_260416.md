## 취약점 목록

### 🔴 P1 — Critical

#### #1 · `api/auth.ts:52-55` — IDOR (타 유저 프로필 수정)

**현상**: PATCH `/api/auth` 엔드포인트에서 `id`를 클라이언트 body에서 받아 서버 검증 없이 UPDATE 실행.

```ts
const { id, display_name, avatar_url } = req.body;
await supabase.from('users').update(updatePayload).eq('id', id);
// id 소유 여부 검증 없음 → 임의 UUID를 전달하면 타인 프로필 수정 가능
```

**영향**: 다른 사용자의 `display_name` / `avatar_url` 덮어쓰기 가능. 익명 앱이어도 세션 사칭 벡터.

**수정 방향**:
- Supabase RLS `auth.uid() = id` 정책 적용 (근본 해결)
- 또는 서버에서 JWT 토큰 검증 후 요청자 ID와 `id` 비교

#### #2 · `api/sessions.ts` — 전 메서드 소유권 검증 부재

**현상**: 모든 HTTP 메서드에서 `user_id` / `session_id`를 쿼리/body에서 그대로 사용.

```ts
// GET: 타인 메시지/세션 열람
.eq('session_id', session_id)   // 소유자 확인 없음
.eq('user_id', user_id)         // 임의 user_id 가능

// DELETE: 타인 세션 삭제
.delete().eq('id', session_id)  // 소유자 확인 없음

// PATCH: 타인 세션 제목 변경
.update({ title }).eq('id', session_id)
```

**영향**: UUID를 알거나 추측할 경우 타인의 대화 내용 열람, 세션 삭제, 제목 변조 가능.

**수정 방향**:
- Supabase RLS `user_id = auth.uid()` 정책이 가장 근본적
- RLS 불가 시 서버에서 `SELECT user_id FROM chat_sessions WHERE id = session_id` 후 요청자와 비교

#### #3 · `api/fetch-url.ts:49` — SSRF (Server-Side Request Forgery)

**현상**: 사용자 제공 URL을 내부 IP 차단 없이 `fetch` 실행.

```ts
const response = await fetch(targetUrl);  // IP 검증 없음
```

**영향**: `http://169.254.169.254/latest/meta-data/` (AWS/GCP 메타데이터), `http://localhost:*`, `http://10.x.x.x` (내부망) 접근 가능.
Vercel 서버리스 환경에서도 클라우드 메타데이터 엔드포인트는 접근 가능.

**수정 방향**:

```ts
const BLOCKED_PATTERNS = [
    /^127\./, /^10\./, /^172\.(1[6-9]|2\d|3[01])\./, /^192\.168\./,
    /^169\.254\./, /^0\.0\.0\.0/, /^localhost$/i, /^::1$/
];

function isPrivateUrl(urlStr: string): boolean {
    try {
        const { hostname } = new URL(urlStr);
        return BLOCKED_PATTERNS.some(p => p.test(hostname));
    } catch { return true; }
}

if (isPrivateUrl(targetUrl)) {
    return res.status(400).json({ error: 'Invalid URL' });
}
```

YouTube 분기에서 line 34의 `pageRes` fetch도 동일 guard 필요.

### Fix A — 약품 카드 이미지 미표시 (모바일)

**현상**: 모바일에서 약품 카드 이미지가 일부 표시 안 됨. ConnectDI URL인 경우 shimmer 계속 표시.

**원인**: `sync-drug-image.ts` 내부 fetch 전체에 timeout 없음 → 모바일 네트워크 지연 시 Vercel 300s까지 hang. `DrugRenderer.tsx` 클라이언트 fetch에도 timeout 없어 shimmer 영구 표시.

**수정 파일 및 내용**:

| 파일 | 추가된 timeout | 대상 |
|------|--------------|------|
| `proxy-image.ts` | 10s | 외부 이미지 프록시 fetch |
| `DrugRenderer.tsx` | 20s | `/api/sync-drug-image` 클라이언트 fetch → timeout 시 abort → proxy fallback 전환 |
| `sync-drug-image.ts` | 5s | Supabase 캐시 HEAD check |
| | 12s/attempt | `fetchWithRetry` 메인 이미지 다운로드 (AbortError 시 재시도 안 함) |
| | 8s × 2 | pharm.or.kr 이미지 fetch (`_b.jpg`, `_s.jpg`) |
| | 8s | Naver/ConnectDI 스크래핑 fetch |
| | 8s × 2 | `fetchConnectDIDetailHTML` HTML fetch + 상세페이지 fetch |

**효과**: ConnectDI URL 기준 20s 내 sync 실패 시 클라이언트가 자동 abort → `setSyncing(false)` → proxy fallback 또는 no-image placeholder로 전환. Shimmer 영구 표시 해소.

### Fix B — 채팅 이전 대화 로드 시 사용자 메시지만 표시

**현상**: 채팅 히스토리 재진입 시 AI 응답 없이 유저 질의만 표시되는 경우 발생.

**원인 분석**:

- 유저 메시지 저장: 스트림 시작 전 fire-and-forget → 스트림 실행 5~30초 동안 완료 ✅
- AI 응답 저장: 스트림 완료 후 fire-and-forget → `res.end()` 직전에 Promise 등록 → **Vercel이 `res.end()` 후 함수 freeze 시 `.then()` callback 미실행** ❌

화면에는 응답이 표시되는 이유: SSE 청크는 스트리밍 도중 이미 브라우저에 전송됨. DB 저장과 무관하게 렌더링됨.

**수정 파일**: `api/chat.ts` — line 279

```ts
// 기존 (fire-and-forget)
supabase.from('chat_messages').insert({...}).then(({ error }) => { ... });
supabase.from('chat_sessions').update({...}).then(({ error }) => { ... });

// 수정 (await — SSE 이미 완료 상태이므로 UX 영향 없음)
try {
    await Promise.all([
        supabase.from('chat_messages').insert({...}),
        supabase.from('chat_sessions').update({...}).eq('id', session_id)
    ]).then(([{ error: msgError }, { error: sessionError }]) => { ... });
} catch (dbError: any) {
    console.error('[Chat API] DB save failed:', dbError.message);
}
```

**효과**: `res.end()` 전에 DB 저장 완료 보장 → 채팅 히스토리 재진입 시 AI 응답 정상 표시.

> **참고**: v4.36(DEV_260413)에서 세션 종료 버그 해결을 위해 도입한 fire-and-forget이 이 문제의 원인. 당시엔 Vercel 60s 한계 회피가 목적이었으나 현재 Fluid Compute에서 기본 timeout이 300s로 증가하여 `await`로 변경해도 무방함.
