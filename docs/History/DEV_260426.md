# DEV_260426 — 스트리밍 중 `**` 볼드 마커 미완성 렌더링 수정

## 배경

AI 응답에서 `**텍스트**` 볼드 강조가 가끔 `**` 기호 그대로 표기되는 문제 발생.

---

## 원인 분석

### 렌더링 구조

`renderContent()` → 텍스트를 viz 블록 기준으로 분할 → 각 텍스트 파트를 독립 `ReactMarkdown` 인스턴스로 렌더링.

### 원인별 분류

| # | 원인 | 빈도 | 수정 가능 여부 |
|---|------|------|--------------|
| 1 | 스트리밍 중 `**` 쌍 미완성 — 닫히는 `**`가 아직 미도착 | **가장 흔함** | ✅ 코드 수정 |
| 2 | viz 블록 기준 텍스트 분할 시 `**` 쌍이 파트 경계로 잘림 | 드묾 | △ 코드 수정 가능 (복잡) |
| 3 | 모델이 ` ** 텍스트 ** ` (앞뒤 공백 포함) 출력 — CommonMark 스펙상 볼드 불인식 | 간헐적 | △ 시스템 프롬프트로 완화 |
| 4 | `maxOutputTokens` 초과로 응답 잘림 — 닫히는 `**` 미도착 | 개선됨 | ✅ 32768으로 이미 상향 |

### 시스템 프롬프트가 최선이 아닌 이유

가장 빈번한 원인(스트리밍 중 쌍 미완성)은 모델 지시로 해결 불가 — 스트리밍 특성 자체의 문제.

### 기존 코드와의 관계

`ChatMessage.tsx`에 이미 동일한 패턴의 처리가 있음:

```ts
// 기존: dangling ``` 처리
const backticks = processedRemaining.match(/```/g);
if (backticks && backticks.length % 2 !== 0) {
    processedRemaining += '\n```';
}
```

`**` 에 대한 동일 처리가 누락된 상태였음.

---

## 수정 내용

**파일:** `components/ChatMessage.tsx`

두 곳(incomplete viz 분기 / normal 분기) 모두에 추가.

```diff
  // Safely close dangling code blocks during streaming
  const backticks = processedRemaining.match(/```/g);
  if (backticks && backticks.length % 2 !== 0) {
    processedRemaining += '\n```';
  }
+ // Close unclosed bold markers during streaming
+ if ((processedRemaining.match(/\*\*/g) || []).length % 2 !== 0) {
+   processedRemaining += '**';
+ }
```

### 동작 방식

스트리밍 도중 `**`의 총 개수가 홀수이면 문자열 끝에 `**`를 추가 → ReactMarkdown이 완전한 쌍으로 파싱 → 볼드 렌더링.
다음 청크 도착 시 실제 닫히는 `**`가 붙으면 appended `**`는 자연스럽게 중화됨.

### 미해결 케이스

- ` ** 텍스트 ** ` (공백 포함): CommonMark 스펙상 볼드 불인식. 빈도 높으면 시스템 프롬프트에 `**텍스트**` 형식(앞뒤 공백 없이) 명시 추가 검토.
- viz 블록 경계 분할: 모델이 `**` 안에 viz 블록을 생성하는 극히 드문 케이스. 현재 감수.

## 수정 2: MFDS 미등재 약품 웹 검색 폴백 개선

### 문제

"제놀한방파스"처럼 파스(패치), 연고, 액제 등 알약·정제가 아닌 의약품은 MFDS 알약식별 DB에 없음.
MFDS 실패 → DuckDuckGo 웹 검색 시도 → DDG 파싱 실패 → 툴이 "식약처 DB와 웹 검색 모두에서 찾지 못했습니다" 반환 → LLM이 이 문장을 읽고 포기 → "찾을 수 없습니다" 출력.

멀티턴에서 사용자가 직접 "검색해줘" 요청 시에는 `general` intent → Google Search grounding → 정상 출력.

### 원인 (2단계)

1. **DuckDuckGo 파싱 실패**: Vercel 서버리스 환경에서 DDG HTML 파싱이 간헐적으로 실패 → 웹 검색 결과 없음
2. **LLM 포기 유도 메시지**: 툴이 "식약처 DB와 웹 검색 **모두** 실패"라고 알려줘서 LLM이 자신의 학습 데이터 사용도 포기

### 수정 내용

**파일 1 (1차 수정 — 시스템 프롬프트 충돌):** `api/_lib/agent/prompt.ts`

- `PROACTIVE DRUG VISUALIZATION` 규칙에 `[MFDS_NOT_FOUND]` 예외 명시
- `PRIORITY RULE`에도 MFDS_NOT_FOUND 예외 추가
- `drug_info` intent focus hint에 `CRITICAL EXCEPTION` 명시

**파일 2 (2~3차 수정):** `api/_lib/agent/drug-info-tool.ts`

MFDS 실패 시 3단계 폴백 체인으로 변경:

```
1. Google Search grounding (Gemini SDK + googleSearch tool) ← 신규, 가장 신뢰성 높음
2. DuckDuckGo (searchWebTool) ← 기존, 파싱 간헐 실패
3. LLM 내부 지식 ← 최후 수단, "찾을 수 없습니다" 금지 지시 포함
```

`searchDrugViaGoogleSearch()` 헬퍼 추가: `GoogleGenAI` SDK로 `gemini-2.5-flash` + `googleSearch` tool 호출, 50자 미만 응답은 실패로 처리. `response.candidates[0].groundingMetadata.groundingChunks`에서 출처 URL 추출 → `[WEB_SOURCE_URLS]` 블록으로 반환 → `chat.ts` `on_tool_end` 핸들러가 소스 칩으로 표시.

**파일 3:** `api/chat.ts`

`on_chat_model_stream` 스트리밍 청크 sanitizer에 누출 방어 추가.

### 2차 수정 — 출처 칩 미표시 (`drug-info-tool.ts` 4차 수정)

**문제:** 내용은 정상 출력되지만 하단 소스 칩이 여전히 미표시.

**원인 분석:**

Gemini `googleSearch` tool 사용 시 `groundingChunks`(개별 웹페이지 URI 목록)는 항상 반환되지 않음. 모델이 검색 결과를 내부적으로만 참조하고 특정 페이지를 명시하지 않는 경우 빈 배열로 반환 → `[WEB_SOURCE_URLS]` 블록 미생성 → `on_tool_end` 핸들러가 소스 없음으로 처리.

`groundingMetadata.webSearchQueries`(Gemini가 실제 사용한 검색어 목록)는 검색 시 거의 항상 반환됨.

**수정 내용:**

- 로그 개선: `chunks: N | queries: [...]` 형태로 grounding 상태를 Vercel 함수 로그에서 확인 가능
- `webSearchQueries` 폴백 추가: `groundingChunks`가 비어 있어도 `webSearchQueries`가 있으면 `https://www.google.com/search?q=...` URL을 `[WEB_SOURCE_URLS]` 블록에 포함 → 소스 칩으로 표시

```ts
const queries = gm?.webSearchQueries as string[] | undefined;
if (queries && queries.length > 0) {
    const searchUrl = `https://www.google.com/search?q=${encodeURIComponent(queries[0])}`;
    return `${text}\n\n[WEB_SOURCE_URLS]\n${searchUrl} | Google 검색: ${queries[0]}`;
}
```

### 3차 수정 — 출처 칩 미표시 근본 해결 (`chat.ts` 5차 수정, `drug-info-tool.ts` 4차 수정)

**문제:** 위 수정 후에도 소스 칩 미표시 지속.

**근본 원인 분석:**

`on_tool_end` 이벤트가 LangGraph `streamEvents` v2 루프에서 실제로 firing되는지, `event.data.output` 구조가 예상과 일치하는지 불확실. `on_tool_end` 의존 자체가 취약한 구조.

**수정 내용 (`api/chat.ts`):**

`on_tool_end`에 의존하지 않고 `on_chain_end` for "LangGraph" 핸들러에서 최종 그래프 상태의 `messages`를 직접 스캔하는 방식으로 변경. 그래프가 완료되면 `data.output.messages`에 모든 ToolMessage가 포함되므로 `[WEB_SOURCE_URLS]` 블록을 확실히 감지할 수 있음.

```ts
const finalMessages: any[] = finalOutput?.messages || [];
for (const msg of finalMessages) {
  const msgType = msg._getType?.() ?? msg.getType?.() ?? msg.type;
  if (msgType === 'tool') {
    const content = typeof msg.content === 'string' ? msg.content : '';
    const urlBlockMatch = content.match(/\[WEB_SOURCE_URLS\]\n([\s\S]+?)(?:\n\n|$)/);
    if (urlBlockMatch) {
      urlBlockMatch[1].split('\n').forEach((line: string) => {
        const [url, ...titleParts] = line.split(' | ');
        const title = titleParts.join(' | ').trim() || url;
        if (url?.startsWith('http') && !allSources.some((e: any) => e.uri === url)) {
          allSources.push({ title, uri: url });
        }
      });
      sendEvent({ sources: allSources });
    }
  }
}
```

진단 로그 `[Chat API] LangGraph end — ToolMessage len: N, hasUrls: true/false`도 추가해 Vercel 로그에서 상태 확인 가능.

### 4차 수정 — 소스 칩 실제 미렌더링 원인 해결 / 스트리밍 완료 후 표시 (`useChatStream.ts`)

**문제:** SSE 응답 데이터 확인 결과 `sources` 이벤트 자체는 텍스트보다 **먼저** 정상 전송되고 있었음. 그러나 UI에는 소스 칩이 여전히 미표시.

**진짜 원인:**

```
data: {"sources":[...]}   ← 먼저 도착
data: {"text":"..."}      ← 이후 도착
```

`sources` 콜백이 실행될 시점에는 `modelMessageId`에 해당하는 메시지가 아직 `session.messages`에 존재하지 않음. `session.messages.map(message => { if (message.id === modelMessageId) {...} })` 조건이 매칭되지 않아 소스가 silently 무시됨. 이후 첫 텍스트 청크로 메시지가 새로 생성될 때는 `manualGroundingSources`만 반영되고 SSE 소스는 누락.

**수정 내용 (`src/hooks/useChatStream.ts`):**

- `pendingSources: any[] = []` 변수 추가
- `sources` 콜백에서 항상 `pendingSources = sources || []`로 최신 소스 보관
- 첫 텍스트로 메시지를 신규 생성할 때 `[...manualGroundingSources, ...pendingSources]`를 `groundingSources`에 반영

```ts
let pendingSources: any[] = [];

// sources 콜백 — 즉시 반영 없이 보관만:
pendingSources = sources || [];

// finally 블록 — 스트리밍 완료 후 일괄 적용:
if (pendingSources.length > 0) {
  setSessions(prev => prev.map(session => {
    if (session.id !== activeSessionId) return session;
    return {
      ...session,
      messages: session.messages.map(message => {
        if (message.id !== modelMessageId) return message;
        const allSources = [...manualGroundingSources, ...pendingSources];
        const uniqueSources = Array.from(new Map(allSources.map(item => [item.uri, item])).values());
        return { ...message, groundingSources: uniqueSources.length > 0 ? uniqueSources : undefined };
      }),
    };
  }));
}
```

**추가 개선 — 소스 칩 스트리밍 중 노출 방지 (5차 수정):**

소스 이벤트가 텍스트보다 먼저 도착하더라도 스트리밍 도중 칩이 나타나는 것이 어색하다는 피드백 반영. `sources` 콜백에서 즉시 `setSessions`를 호출하지 않고 `pendingSources`에 보관만 한 뒤, `finally` 블록(스트리밍 완료 후)에 일괄 적용. `general` intent Google Search grounding 소스도 동일 경로를 타므로 함께 적용됨.

### 구조적 한계

파스·연고·시럽 등 비알약 제형은 MFDS 알약식별 DB에 원천 미등재 → DrugRenderer 카드 없이 마크다운으로만 안내. DDG 파싱 실패 시 LLM 학습 데이터 기반 답변 (잘 알려진 OTC 제품은 대부분 커버).

## 수정 4: 스트리밍 `[N]` citation 청크 분할 수정

### 문제

PDF 문서 분석 시 응답 문장 끝에 `[15]` 등 인용 마커가 남는 경우 발생. 기존 per-chunk strip은 `[15]`가 두 청크로 나뉘면 동작 안 함:
- 청크 A: `"...문장 ["` → 패턴 미매칭 → `[` 그대로 전송
- 청크 B: `"15] 다음"` → 패턴 미매칭 → `15]` 그대로 전송

### 수정 내용

**`api/_lib/agent/nodes/generator.ts`** (SDK 스트리밍 — general intent, PDF 분석)
- `citationBuffer` 추가: 청크 처리 시 이전 버퍼 + 현재 청크 합산 후 strip
- `/\s?\[\d*(?:,\s*\d*)*$/` 로 끝 불완전 패턴 검출 → 버퍼 보류
- 스트림 종료 후 버퍼 flush (불완전 패턴 = 진짜 인용 아님)

**`api/chat.ts`** (LangChain `on_chat_model_stream` — drug_id/drug_info)
- `lcCitationBuffer` 동일 패턴 적용
- `on_chain_end(LangGraph)` 이벤트에서 flush

## 수정 5: 첨부파일 UX 개선 — 직사각형 레이아웃 & 문서 카드 정보 밀도

### 배경

기존 첨부파일 표시가 이미지는 정사각형, 문서는 파일명 잘림이 심하고 파일 크기 정보가 없어 부자연스럽다는 피드백.

### 변경 사항

**파일 1: `types.ts`**

`MessageAttachment`에 `fileSize?: number` 필드 추가.
- 옵셔널이므로 기존 localStorage 저장 세션과 완전 하위 호환.

**파일 2: `components/ChatInput.tsx`**

입력창 썸네일 전면 개편 (전송 전 미리보기):

| 구분 | 이전 | 이후 |
|------|------|------|
| 이미지 | `h-16 w-16` 정사각형 | `128×72px` 직사각형, 확장자 배지 우하단 |
| 비디오 | `h-16 w-16` 정사각형 | `128×72px` 직사각형, 재생 아이콘 + 배지 |
| 문서 | `h-16 w-32`, 아이콘 세로 배치 | `160×72px`, 아이콘 + 파일명 + 확장자 가로 배치 |
| 삭제 버튼 | 컨테이너 외부 `-top-2.5 -right-2.5` 원형 버튼 | hover 시 전체 오버레이 X (클리핑 이슈 해결) |

- `processFile`에서 `file.size`를 `attachment.fileSize`로 저장.

**파일 3: `components/ChatMessage.tsx`**

채팅 기록 첨부파일 카드 개선 (전송 후 이력):

1. **이미지**: `h-auto` 자연 비율 → `h-200px` 고정 직사각형. 클릭 핸들러를 이미지가 아닌 감싸는 div로 이동해 `cursor-pointer` 적용.
2. **문서 카드** (PDF/docx/xlsx/pptx/csv/hwpx):
   - 아이콘 컨테이너 `w-10 h-10` → `w-12 h-12`, `rounded-lg` → `rounded-xl`
   - 아이콘 크기 `text-xl` → `text-2xl`
   - 아이콘 배경에 얇은 테두리 추가 (`border` + 타입별 색상)
   - 파일명 너비 `max-w-[200px]` → `max-w-[280px]` (긴 이름 잘림 완화)
   - 파일 크기 `formatFileSize()` 헬퍼로 `· 2.4 MB` 형태 표시
   - 타입별 hover 색상 추가 (PDF: `hover:border-red-300`, docx: 파란색, xlsx: 초록색 등)
   - 전체 카드 `rounded-xl` → `rounded-2xl`
3. **`AttachmentImage` 컴포넌트**: `style?: React.CSSProperties` prop 추가 (이미지 직사각형 높이 고정에 필요).
4. **`formatFileSize()` 헬퍼**: bytes → `KB` / `MB` 포맷 변환. `fileSize` 없는 기존 이력은 크기 미표시로 graceful 처리.

### 미적용 사항 (논의 후 보류)

- 이미지 + 텍스트 단일 버블 통합: DOM 구조 전면 리팩토링 필요, 별도 작업으로 이월.
- 다중 문서 2컬 그리드: 임팩트 대비 복잡도가 높아 보류.

## 수정 6: 이미지 썸네일 비율/크기 튜닝 & 히스토리 미리보기 복원

### 배경

수정 5에서 적용한 `h-200px` 고정 높이가 세로로 너무 길게 표시되고, 이후 `h-auto + maxHeight: 280px`로 수정했으나 이번엔 너무 짧게 표시되는 문제. 이미지 로드 실패(IMG 폴백) 시에도 컨테이너가 찌그러지는 이슈 병행 발생.

또한 소용량 이미지(<3MB)는 세션 로드 시 base64 복원이 불가하여 히스토리에서 항상 IMG 폴백만 표시되는 구조적 문제 확인.

### 이미지 썸네일 컨테이너 개선

**파일: `components/ChatMessage.tsx`**

| 항목 | 변경 전 | 변경 후 |
|------|---------|--------|
| 컨테이너 구조 | `h-auto + maxHeight` | `aspect-ratio: 16/9` 고정 비율 컨테이너 |
| 최대 너비 | `max-w-[420px]` | `max-w-[220px]` (220×124px) |
| 이미지 fill | `object-cover h-auto` | `w-full h-full object-cover` |
| 폴백 컨테이너 | `${className}` (크기 비결정) | `w-full h-full` (부모 비율 따름) |
| 폴백 아이콘 | `text-3xl`, `text-xs` | `text-lg`, `text-[10px]` |
| 다중 이미지 그리드 | `h-32 sm:h-48` 고정 높이 | 동일 `aspect-ratio: 16/9` 적용 |

**효과:**
- 이미지 원본 비율과 무관하게 항상 일관된 16:9 직사각형 썸네일
- 이미지 로드 실패(폴백) 시에도 동일 크기의 placeholder 박스 표시
- ChatGPT / Telegram / iMessage 방식과 동일한 고정 비율 썸네일 UX

### 이미지 항상 Supabase 업로드

**파일: `src/hooks/useChatStream.ts`**

**문제:**
`chat_messages.attachment_url` 컬럼에 소용량 이미지는 `"image/jpeg"` mimeType 문자열만 저장되고 base64 데이터는 미저장. 세션 로드 시 `inferAttachment()`가 `data: ''`를 반환 → IMG 폴백만 표시.

**변경:**

```diff
- if (!isVideo && estimatedSize < (3 * 1024 * 1024) && isBase64) {
+ // 이미지: 크기 무관하게 항상 Supabase 업로드 (히스토리 미리보기 복원)
+ // 문서: 3MB 미만 base64 inline 허용
+ if (!isImage && !isVideo && estimatedSize < (3 * 1024 * 1024) && isBase64) {
```

- **이미지**: 크기에 무관하게 항상 `chat-imgs` 버킷에 업로드 → Supabase public URL을 DB에 저장
- **문서(PDF/docx 등)**: 기존대로 3MB 미만은 base64 inline 유지
- **비디오**: 기존대로 항상 업로드

**결과:**
- 전송 시: 이미지 업로드 (~0.5~2s) 후 전송 ("파일명 업로드 중..." 상태 표시)
- 히스토리 로드 시: Supabase CDN URL로 이미지 정상 표시
- 기존 세션의 소용량 이미지: base64가 DB에 미저장으로 복원 불가 (IMG 폴백 유지)
