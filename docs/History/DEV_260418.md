## 1. 웹 요약 버그 수정

### 문제
URL 요약 첫 시도 시 아무것도 표시되지 않다가 두 번째 시도에서 정상 동작하는 현상.

### 근본 원인

`fetch-url` 타임아웃(10s 서버 / 15s 클라이언트) 또는 네트워크 에러 발생 시:

1. `fetchUrlContent`가 에러를 catch하여 `""` 또는 `[FETCH_ERROR: AbortError]` 반환
2. `useChatStream.ts`에서 `[URL_CONTENT: url]\n` 태그가 내용 없이 webContext에 붙음
3. `generator.ts`에서 `hasUrlContent = state.webContent.includes('[URL_CONTENT]')` → **true**
4. Google Search가 비활성화됨 (`useGoogleSearch = false`)
5. LLM이 빈/에러 내용으로 호출 → 빈 응답 → `receivedAnyText = false` → 에러 표시
6. 두 번째 시도는 Vercel warm + 페이지 응답 빠름 → fetch-url 성공 → 정상 동작

추가로 `urlFetchError` 플래그가 절대 세워지지 않는 문제 (`fetchUrlContent`가 throw하지 않음).

### 수정 내용

**`services/geminiService.ts`**
```ts
// fetchUrlContent: [FETCH_ERROR] 반환값을 빈 문자열로 처리
export const fetchUrlContent = async (url: string): Promise<string> => {
  const data = await fetchUrlData(url);
  if (!data.content || data.content.startsWith('[FETCH_ERROR')) return '';
  return data.content;
};
```

**`src/hooks/useChatStream.ts`**
- pageContent가 빈 문자열이면 `[URL_CONTENT]` 태그를 webContext에 붙이지 않음
- URL fetch 실패 시 에러 메시지를 사용자에게 표시하지 않음 (Google Search가 자동 대체)

```ts
const pageContent = await fetchUrlContent(url);
if (pageContent) {
  webContext += `\n\n[URL_CONTENT: ${url}]\n${pageContent}`;
} else {
  // Google Search가 자동으로 대체 — 에러 표시 없음
}
```

**`api/_lib/agent/nodes/generator.ts`**
- `hasUrlContent` 체크를 정확하게 개선 — 태그만 있고 실제 내용이 없으면 false

```ts
// Before (버그)
const hasUrlContent = state.webContent.includes('[URL_CONTENT]');

// After (수정)
const urlContentMatch = state.webContent.match(/\[URL_CONTENT:[^\]]+\]\n([\s\S]+)/);
const hasUrlContent = !!(urlContentMatch && urlContentMatch[1].trim());
```

**효과**: URL fetch 실패 시 Google Search 유지 → 첫 시도에서도 관련 정보 응답 가능

## 2. Lighthouse 성능 분석 (점수: 70점)

> 수정 미완료 — 향후 작업 예정

### 현재 지표

| 지표 | 현재값 |
|------|--------|
| Performance | 70 |
| FCP | 1.1s |
| LCP | 1.5s |

| CLS | 0 |

### 원인 분석

#### 1. TBT 540ms — Forced Reflow (최대 감점 원인)
- `index-FRUM-R8l.js:11`, `:300`, `:312`에서 forced reflow 발생
- JS가 DOM 스타일 변경 후 `offsetWidth` 등 레이아웃 속성을 즉시 읽어 layout thrashing 발생
- 소스맵 없이 minified 번들에서 정확한 위치 추적 어려움

**개선 방법**:
- `vite.config.ts`에서 `sourcemap: true`로 설정 후 Chrome DevTools Performance 탭으로 원인 추적
- 의심 위치: MutationObserver(`App.tsx`), framer-motion 애니메이션, 렌더 중 레이아웃 읽기

#### 2. ChatMessage 레이지 로딩 워터폴
- `/api/sessions` (2,454ms) 응답 후 ChatMessage 청크 로딩 시작 → 순차 지연
- 이전 커밋(40b2806)에서 lazy 전환이 역효과 발생
- 세션 응답 → ChatMessage 청크 로딩 → 렌더 순서로 waterfall 형성

**개선 방법 (선택)**:
```ts
// useChatSessions.ts — 세션 fetch와 동시에 ChatMessage prefetch
import('./components/ChatMessage'); // side-effect only
```
또는 `App.tsx`에서 ChatMessage를 다시 eager import로 되돌림.

#### 3. 미사용 JS 281KB
- Index bundle: 636.8KB 전송, 191.9KB 미사용
- Markdown bundle: 313.7KB 전송, 89.6KB 미사용

**개선 방법**:
- markdown-vendor 청크는 ChatMessage가 실제 렌더될 때만 로드되어야 함
- `manualChunks`에서 markdown 라이브러리를 제거하고 동적 import로 처리:
```ts
// ChatMessage.tsx 내부
const ReactMarkdown = lazy(() => import('react-markdown'));
```

#### 4. FontAwesome CSS 14.3KB 전부 미사용
- `fontawesome.min.css`가 cdnjs에서 로드되지만 Lighthouse가 미사용으로 감지
- `index.html`에서 직접 `@font-face` 정의 + woff2 프리로드로 대체 가능

**개선 방법**:
```html
<!-- index.html — fontawesome.min.css 제거, @font-face만 유지 -->
<!-- solid.min.css, regular.min.css → 인라인 필요 아이콘만 SVG로 대체 고려 -->
```

#### 5. /api/sessions Supabase Cold Start (2,454ms)
- 첫 로드 시 Supabase 함수 cold start로 세션 API 지연
- localStorage 캐시(`chat_sessions_cache_v1`)가 있어 실제 사용자 경험은 덜 영향
- Lighthouse는 항상 cold start 시뮬레이션 → 점수에 반영됨

**개선 방법**:
- Vercel Edge Config 또는 KV로 세션 목록 캐싱 (cold start 우회)
- 또는 세션 API를 non-blocking으로 처리하여 critical path에서 제외
