## 모바일 YouTube 요약 연결 끊김 개선 기획

### 현상

- 데스크탑: YouTube URL 입력 → 자막 fetch 성공 → 텍스트 요약 정상 동작
- 모바일: YouTube URL 입력 → 자막 fetch 실패 → Native Video Analysis 폴백 → 연결 끊김

---

### 근본 원인 분석

#### 실패 체인

```
모바일 사용자 YouTube URL 입력
    ↓
[프론트] fetchYoutubeTranscript(videoId)
    → POST /api/fetch-transcript (Edge runtime, 최대 30s)
         ├─ YouTube HTML 페이지 fetch: AbortController 10s 타임아웃
         │    └─ 모바일 느린 네트워크에서 2~5MB 페이지를 10s 내 못 받음 → AbortError
         └─ XML 자막 fetch: AbortController 8s 타임아웃

    ↓ 자막 실패 → transcript: null 반환

[백엔드] api/chat.ts — hasTranscript === false이므로 폴백 실행:
    humanMessageParts.push({ fileData: { fileUri: videoUrl, mimeType: 'video/mp4' } })
    → Gemini가 YouTube 영상 전체 다운로드 + 분석 (30~120초 소요)

    ↓ Vercel maxDuration: 60s 초과 → 스트림 강제 종료 → 연결 끊김
```

#### 데스크탑과 차이나는 이유

| 항목 | 데스크탑 | 모바일 |
|------|---------|--------|
| 네트워크 지연 | 50~100ms | 200~500ms |
| 다운로드 속도 | 10Mbps+ | 2~5Mbps (4G) |
| YouTube HTML (2~5MB) fetch 시간 | 1~3s → 10s 내 성공 | 5~15s → 10s 타임아웃 초과 |
| 자막 fetch 결과 | 성공 → 텍스트 요약 | 실패 → native video 폴백 |
| Gemini 처리 시간 | 5~20s (텍스트) | 30~120s (영상 분석) |
| Vercel 60s 제한 영향 | 거의 없음 | 빈번히 초과 |

#### 관련 파일 및 코드 위치

### 개선 방안

#### 1순위 — 자막 fetch 안정화 (핵심)

**`api/fetch-transcript.ts`**

- `runtime: 'edge'` 제거 → Node.js 런타임으로 전환
  - Edge 런타임은 Vercel 30s 하드캡 존재, Node.js는 maxDuration까지 사용 가능
- HTML 페이지 fetch 타임아웃: 10s → **25s**
- XML 자막 fetch 타임아웃: 8s → **15s**
- 총 최대 소요 시간: ~40s (60s 이내)

```ts
// 변경 전
export const config = { runtime: 'edge' };
const pageTimeout = setTimeout(() => pageController.abort(), 10000);
const xmlTimeout = setTimeout(() => xmlController.abort(), 8000);

// 변경 후
// config 블록 삭제 (Node.js 기본 런타임 사용)
const pageTimeout = setTimeout(() => pageController.abort(), 25000);
const xmlTimeout = setTimeout(() => xmlController.abort(), 15000);
```

#### 2순위 — Vercel maxDuration 증가

**`vercel.json`**

- 60s → 300s (Vercel Pro 플랜 기준 최대 300s)
- native video analysis 폴백이 실행되더라도 완료 가능

```json
{
  "functions": {
    "api/*.ts": {
      "maxDuration": 300
    }
  }
}
```

> 현재 플랜 확인 필요: Hobby 플랜은 최대 60s, Pro 이상에서 300s 가능

#### 3순위 — SSE 연결 안정성 강화

**`api/chat.ts`**

- Heartbeat 간격 단축: 15s → **8s**
  - 모바일 네트워크/프록시에서 idle 연결 드롭 방지
- 모바일 프록시 버퍼링 방지 헤더 추가:

```ts
// 변경 전
res.setHeader('Content-Type', 'text/event-stream');
res.setHeader('Cache-Control', 'no-cache');
res.setHeader('Connection', 'keep-alive');

// 변경 후
res.setHeader('Content-Type', 'text/event-stream');
res.setHeader('Cache-Control', 'no-cache');
res.setHeader('Connection', 'keep-alive');
res.setHeader('X-Accel-Buffering', 'no');  // nginx 프록시 버퍼링 비활성화
```

#### 4순위 — 프론트엔드 타임아웃 명시

**`services/geminiService.ts`**

`fetchYoutubeTranscript` 호출 시 AbortController로 명시적 타임아웃 설정 (현재 없음):

```ts
export const fetchYoutubeTranscript = async (videoId: string): Promise<string | null> => {
  const controller = new AbortController();
  const timeout = setTimeout(() => controller.abort(), 45000); // 45s
  try {
    const response = await fetch('/api/fetch-transcript', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ videoId }),
      signal: controller.signal,
    });
    const data = await response.json();
    return data.transcript || null;
  } catch (error) {
    return null;
  } finally {
    clearTimeout(timeout);
  }
};
```

## 실제 수정 내역 (2026-04-20)

### `api/fetch-transcript.ts`
- `export const config = { runtime: 'edge' }` 블록 삭제 → Node.js 런타임으로 전환 (Edge 30s 하드캡 → Node.js 60s)
- HTML 페이지 fetch AbortController 타임아웃: `10000` → `25000` (25s)
- XML 자막 fetch AbortController 타임아웃: `8000` → `15000` (15s)
- 총 최대 소요 시간: ~40s → Vercel 60s 이내 안정적으로 완료 가능

### `api/chat.ts`
- SSE 헤더에 `X-Accel-Buffering: no` 추가 → nginx/모바일 프록시 응답 버퍼링 방지
- Heartbeat 간격: `15000` → `8000` (8s) → 모바일 idle 연결 드롭 방지

### `services/geminiService.ts`
- `fetchYoutubeTranscript`에 AbortController 기반 45s 타임아웃 추가
- fetch 요청에 `signal: controller.signal` 적용 + `finally` 블록에서 `clearTimeout` 보장

### `src/hooks/useChatStream.ts`
- **자동 1회 재시도 로직 추가**: Vercel cold start로 인해 스트림이 텍스트 없이 종료되는 경우(`응답을 받지 못했습니다` / `LLM returned empty response`) 에러를 유저에게 노출하지 않고 자동 재시도
- 재시도 시 이전 부분 응답 초기화 + `재시도 중...` 상태 표시 후 1.5s 대기 (Vercel warm-up 시간 확보)
- 재시도 불가 에러(429, 503, 인증 오류 등)는 즉시 에러 표시로 유지
