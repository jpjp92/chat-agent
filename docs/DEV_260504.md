## 1. 보안 수정 (Security Hardening)

### 1-1. SSRF 블록리스트 확장 — `api/fetch-url.ts`, `api/proxy-image.ts`

기존 블록리스트에 RFC 1918 사설 IP 대역 및 IPv6 사설 대역 전체 추가.

```
기존: localhost, 127.x, 0.0.0.0, 169.254.x (AWS 메타데이터), ::1
추가: 10.x/8, 172.16-31.x/12, 192.168.x/16, fc::/7, fd::/8, fe80::
```

```ts
const blocked = /^(localhost|127\.\d+\.\d+\.\d+|0\.0\.0\.0|10\.\d+\.\d+\.\d+|172\.(1[6-9]|2\d|3[01])\.\d+\.\d+|192\.168\.\d+\.\d+|169\.254\.\d+\.\d+|::1|fc[\da-f]{2}:|fd[\da-f]{2}:|fe80:)/i.test(hostname);
```

### 1-7. CSP 헤더 추가 — `vercel.json`

```json
{ "key": "Content-Security-Policy", "value": "frame-src 'none'; object-src 'none'; base-uri 'self'" }
```

- `frame-src 'none'`: clickjacking 방어 (기존 `X-Frame-Options`과 이중 방어)
- `object-src 'none'`: Flash 등 플러그인 실행 차단
- `base-uri 'self'`: base tag injection 차단
- `default-src`, `script-src` 미설정 (React 앱 동작에 영향 최소화)

## 2. 에러 메시지 다국어 처리 (i18n Error Handling)

> 지원 언어: `ko` | `en` | `es` | `fr`

### 2-1. `api/chat.ts` — LLM 에러 메시지 다국어화 (가장 영향 범위 큼)

`language` 파라미터가 `req.body`로 이미 수신되고 있었으나 에러 분기에서 미사용.  
`CHAT_ERRORS` 맵을 핸들러 외부(모듈 레벨)에 정의하고 catch 블록에서 `language`로 선택.

```ts
const CHAT_ERRORS: Record<string, Record<string, string>> = {
  rateLimit:   { ko: '요청이 많아...', en: 'Too many requests...', es: '...', fr: '...' },
  unavailable: { ko: '서버가 일시적으로...', en: 'Server temporarily...', es: '...', fr: '...' },
  auth:        { ko: '인증 오류...', en: 'Authentication error...', es: '...', fr: '...' },
  generic:     { ko: '응답 생성 중...', en: 'Failed to generate...', es: '...', fr: '...' },
};

// catch 블록
const lang = (['ko', 'en', 'es', 'fr'].includes(language)) ? language : 'ko';
const errorType = status === 429 || ... ? 'rateLimit' : ... : 'generic';
sendEvent({ error: CHAT_ERRORS[errorType][lang] });
```

## 4. 잔여 미수정 항목

### 4-1. IDOR (High)
`api/sessions.ts` — `user_id`, `session_id`를 클라이언트에서 받아 검증 없이 쿼리 사용.  
API 레벨 수정만으로는 근본 해결 불가. **Supabase RLS 정책 설계 별도 필요.**

### 4-2. CSP `default-src` 미설정 (Low)
현재 `frame-src`, `object-src`, `base-uri` 3개만 설정.  
`script-src`, `connect-src` 추가 시 React 번들, Gemini API 엔드포인트, Supabase URL 등 전수 파악 필요. 별도 작업으로 진행.

### 4-3. `sync-drug-image.ts` ALLOWED_DRUG_HOSTS 확장 필요 시 주의 (관리 항목)

현재 화이트리스트:
```ts
const ALLOWED_DRUG_HOSTS = ['nedrug.mfds.go.kr', 'pstatic.net', 'connectdi.com', 'terms.naver.com', 'health.kr'];
```

**각 도메인의 역할:**

| 도메인 | 역할 |
|--------|------|
| `nedrug.mfds.go.kr` | MFDS 식약처 공식 이미지 (1순위) |
| `connectdi.com` | ConnectDI 의약품 DB 폴백 (2순위) |
| `health.kr` | AI가 생성하는 hybrid URL 패턴 (`health.kr/https://pstatic.net/...`) 진입점 |
| `pstatic.net` | URL 오토리페어 후 최종 도달하는 Naver CDN |
| `terms.naver.com` | Naver 지식백과 약품 페이지 |

**향후 약품 데이터 소스 추가 시 필수 업데이트:**  
KIMS(`kimsonline.co.kr`), 드럭인포(`druginfo.mfds.go.kr`) 등 새 DB를 연동하면  
해당 이미지 도메인을 `ALLOWED_DRUG_HOSTS`에 추가하지 않으면 이미지 sync가 `400`으로  
**조용히 실패**하고 DrugRenderer가 proxy fallback으로 넘어간다.  
에러 로그: `[Sync] Blocked SSRF attempt to: <hostname>` 으로 확인 가능.
