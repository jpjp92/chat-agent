# DEV_260507.md

## 1. 프론트엔드 대용량 문서 페이로드(500 에러) 원천 차단
- **문제**: 사용자가 3MB에 육박하는 문서를 업로드할 경우, 기존 `useChatStream.ts` 로직 상 3MB 미만은 Base64로 인코딩되어 Vercel 서버리스 페이로드 본문에 포함됨. 이로 인해 Vercel의 4.5MB 페이로드 제한을 초과하여 `500 Internal Server Error` 발생 가능.
- **해결**: 문서 인라인(Base64) 허용 임계값을 3MB에서 `1MB`로 대폭 축소(`estimatedSize < (1 * 1024 * 1024)`). 1MB 이상의 문서는 즉시 Supabase `chat-docs` 버킷에 업로드되고, 서버로는 가벼운 URL 형태만 전송되도록 우회 로직 강화. (대용량 파일 처리 시 Vercel 뻗음 현상 영구 종식).

## 2. 이미지 멀티턴 메모리 한계(Google Search 딜레마) 검토 및 폐기 (Wontfix)
- **문제**: TODO 3순위 과제로 "이미지 업로드 후 3턴이 지나면 LLM이 이미지를 까먹는 현상"을 해결하려 시도.
- **실험 및 원인 규명**: `chat.ts`에서 이미지를 컨텍스트에서 탈락하지 않게 예외(`hasImage -> isRecent = true`)를 추가해 보았으나, 치명적인 부작용 발견. **Gemini API는 프롬프트 내에 멀티모달 데이터(이미지)가 존재하면 구글 실시간 검색(Grounding) 기능을 강제로 완전 차단**함. 따라서 대화 내내 이미지를 기억하게 만들면, 해당 세션에서는 영원히 날씨 등 실시간 검색을 사용할 수 없는 상태(봉인)에 빠짐.
- **결정**: "과거 이미지를 기억하는 것"보다 "대화가 진행되면 실시간 구글 검색 기능을 다시 쓸 수 있게 복구되는 것"이 훨씬 중요한 핵심 기능(UX)으로 판단. 해당 이미지 기억 보존 코드는 즉시 **원상 복구(Revert)**하고, TODO의 I4 과제는 **기술적 한계로 인한 폐기(Wontfix)** 처리 완료.
