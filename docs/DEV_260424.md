# DEV_260424 — SDK 스트리밍 경로 인라인 인용 마커 [N] 미제거 버그 수정

## 배경

URL 기반 요약 시 응답 텍스트에 `[1]`, `[2]` 등 인라인 인용 마커가 간헐적으로 남는 문제 발생.
출처는 하단 chip UI로 표시되므로 인라인 마커는 제거되어야 하는데, SDK 스트리밍 경로에서 누락됨.

---

## 원인 분석

### 경로별 [N] 제거 처리 비교

| 경로 | 처리 위치 | 동작 |
|------|-----------|------|
| LangChain (drug_id, drug_info) | `chat.ts:221` `on_chat_model_stream` | 청크 전송 **전** 제거 ✓ |
| SDK 스트리밍 (일반·URL 요약 등) | `generator.ts:271-277` (구 코드) | `AIMessage` 수정 — 클라이언트 전송 **후** ✗ |
| SDK 비스트리밍 fallback | `generator.ts:260` (구 코드) | `sendEvent` 전송 **후** strip ✗ |

### 구체적 버그 흐름 (SDK 스트리밍)

```
generator.ts (SDK 스트리밍 루프)
  └─ sendEvent({ text: sanitized })   ← [N] 포함 상태로 클라이언트 전송
  
  … 루프 종료 후 …
  if (!useGoogleSearch) {
      responseText = responseText.replace(...)  ← AIMessage 상태만 수정, 클라이언트 무관
  }
```

`chat.ts`의 `trackingEvent`는 `sendEvent(data)`를 그대로 호출하므로 strip 기회 없음.
`on_chat_model_stream` (strip 적용)은 LangChain 경로 전용 이벤트 — SDK 경로에서 발화되지 않음.

### 발생 시나리오

1. **URL 본문 추출 성공 (≥300자) + `useGoogleSearch=false`**: 모델이 자체적으로 `[1]` 스타일 마커를 생성할 때 스트림에 그대로 전달.
2. **URL fetch 실패/짧음 → `useGoogleSearch=true`**: Google Search grounding이 `[1]`, `[2]` 마커를 자동 삽입하는데, SDK 청크 전송 시 strip 없음.
3. **비스트리밍 fallback 경로**: `sendEvent({ text: responseText })` 호출 후에 strip이 실행되어 클라이언트에는 마커가 포함된 텍스트 전달.

## 버그 분석: 새 세션 첫 질의 시 스피너 미표시

### 증상

웹에서 새 채팅 세션 시작 후 첫 번째 질의를 보내면 응답이 도착할 때까지 bouncing dot 스피너가 표시되지 않음.
두 번째 질의부터는 정상 표시. 응답 자체는 정상.

### 원인 분석

**복합 원인 2가지가 겹침**

**원인 1 — `createSession()` 대기 중 스피너 조건 미충족**

`ChatArea.tsx:46`:
```tsx
{((isTyping && messages.length > 0 && messages[messages.length - 1].role === Role.USER) || loadingStatus) && (
```

신규 세션 생성 흐름:
```
L74: setIsTyping(true)
     ↓ 하지만 messages=[] (아직 setSessions 전)
     → messages.length > 0 = false → 스피너 조건 실패 ❌
L76: await createSession()  ← 이 대기 시간 동안 스피너 없음
```

`createSession()` 완료 후 `setSessions([newSession])` 으로 userMessage가 추가되고
나서야 스피너 조건을 충족할 수 있는 구조.

**원인 2 — B1 수정의 부작용: useEffect가 새 세션 생성을 세션 전환으로 오인**

`useChatStream.ts` B1 수정 (세션 전환 시 isTyping 리셋):
```ts
useEffect(() => {
  setIsTyping(false);
  setLoadingStatus(null);
}, [currentSessionId]);
```

신규 세션 생성 시에도 `setCurrentSessionId(session.id)` (null → id) 가 호출되어
이 useEffect가 발화 → **스피너가 잠깐 보이다가 즉시 사라짐**.

발생 흐름:
```
L91: setSessions([newSession])        ← userMessage 추가
L92: setCurrentSessionId(session.id) ← null → id 전환
L113: setIsTyping(true)
      ↓ React 배치 렌더 → isTyping=true, messages=[userMsg]
      → 스피너 조건 충족, 잠깐 표시 ✅

useEffect([currentSessionId]) 발화 ← B1 수정의 부작용
  → setIsTyping(false) ← 스피너 즉시 사라짐 ❌

스트리밍 시작 시 isTyping=false 상태 → 스피너 없이 응답 텍스트만 나타남
```

### 해결 방안

**`prevSessionIdRef`로 전환 유형 구분**

`null → sessionId` 전환은 새 세션 생성(handleSendMessage 내부) → useEffect 리셋 skip.
`sessionId → anything` 전환은 사용자 세션 전환 → isTyping/loadingStatus 리셋.

| 전환 유형 | prev | current | 동작 |
|-----------|------|---------|------|
| 새 세션 생성 | null | sessionId | skip (스트리밍 진행 중) |
| 세션 간 전환 | sessionIdA | sessionIdB | 리셋 ✅ |
| 새 채팅 클릭 | sessionId | null | 리셋 ✅ |
| 앱 첫 로드 복원 | null | sessionId | skip (isTyping=false 초기값 유지) ✅ |

## 개선 계획: 멀티턴 응답 잘림 방지 + 대화 길이 제한

### 현황 및 문제점

**원인 1 — `maxOutputTokens: 8192` 하드캡 (주된 원인)**

`api/_lib/agent/nodes/generator.ts` SDK 스트리밍(L188) / fallback(L248) / LangChain(L337) 세 경로 모두 `maxOutputTokens: 8192`.
Gemini 2.5 Flash 실제 최대 출력은 65,536 tokens인데 12.5%만 허용 중.
한국어는 토크나이저 효율이 낮아 8,192 tokens ≈ 약 3,500~5,000자 수준.
긴 코드·상세 분석·멀티파트 응답에서 mid-sentence 잘림 발생.

```ts
// generator.ts — MAX_TOKENS 감지는 하지만 사용자 알림·재시도 없음
if (finishReason === 'MAX_TOKENS') {
    console.warn('...');  // 경고 로그만, 클라이언트에 전달 안됨
}
```

**원인 2 — 멀티턴 history 누적으로 입력 비대화**

`api/chat.ts:53` `.slice(-10)` — 최근 10개 메시지만 전송.
10개 안에 포함된 이전 AI 응답들이 길수록(코드, 분석) 입력 토큰 누적.
모델이 이전에 상세히 답한 컨텍스트가 많을수록 새 응답을 더 짧게 압축하려는 경향.

**원인 3 — `history.slice(-10)` 컨텍스트 탈락**

20개 메시지 시점부터 오래된 컨텍스트의 절반 이상이 API 전송에서 누락됨.
사용자는 이를 모르고 이전 대화 기반 질의를 계속 → 일관성 없는 응답.

---

### 개선 방향

#### Fix 1 — `maxOutputTokens` 상향 (즉각 적용)

| 항목 | 현재 | 변경 |
|------|------|------|
| SDK 스트리밍 (L188) | 8,192 | 32,768 |
| Fallback generateContent (L248) | 8,192 | 32,768 |
| LangChain path (L337) | 8,192 | 32,768 |

32,768 = Gemini 2.5 Flash 최대(65,536)의 절반. 대부분의 긴 응답을 커버하면서 API 비용 폭증 방지.

#### Fix 2 — 멀티턴 경고 + 차단 (프론트엔드)

**임계값 설계:**

| 단계 | 메시지 수 | 기준 | 동작 |
|------|-----------|------|------|
| 경고 | 20개 (10턴) | `slice(-10)` 기준 절반 누락 시작 | Toast `warn` 표시 |
| 차단 | 30개 (15턴) | 컨텍스트 대부분 누락 | 전송 차단 + 인라인 배너 |

**구현 위치**: `src/hooks/useChatStream.ts` `handleSendMessage` 진입부.
백엔드 왕복 없이 `activeSession.messages.length`로 즉시 판단.

```
handleSendMessage 진입
  ├─ messages.length >= 30 → onLimitReached() 콜백 → return (전송 차단)
  ├─ messages.length === 20 → showToast('warn', ..., duration=6000)
  └─ 정상 처리
```

**경고 Toast (20개 도달 시):**
- duration: 6,000ms
- `Toast.tsx`에 `'warn'` 타입 추가 (앰버/옐로우 계열)
- 문구: "대화가 길어지고 있습니다. 새 채팅을 시작하면 더 정확한 응답을 받을 수 있습니다"

**차단 배너 (30개 도달 시):**
- `ChatArea.tsx` 하단에 인라인 배너 — duration 없이 고정 표시
- "새 채팅 시작" 버튼 포함 → `setCurrentSessionId(null)` 호출
- `isLimitReached: boolean` prop으로 ChatArea에 전달

**i18n:**
기존 `statusMessages` 외부 주입 패턴 동일하게 적용.
```ts
interface ChatStreamMessages {
  // 기존...
  turnLimitWarning: string;
  turnLimitReached: string;
}
```
