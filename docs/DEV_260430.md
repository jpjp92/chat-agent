# YouTube 요약 처리 성능 개선 (2026-04-30 — 완료)

## 배경

YouTube URL 요약/분석 요청 시 응답까지 체감 대기 시간이 길다는 문제 제기.  
전체 요청 파이프라인을 검토한 결과 4개 병목 지점 확인 → 3개 파일 수정으로 개선.

---

## 파이프라인 구조 (개선 전)

```
[클라이언트]
  ① fetchUrlContent(url)         → 메타데이터 (직렬)
  ② fetchYoutubeTranscript(id)   → 자막 fetch (직렬, 클라이언트 timeout: 45s)

[서버 /api/fetch-transcript]
  ③ YouTube 페이지 HTML fetch    (timeout: 12s, 직렬)
  ④ caption XML fetch            (timeout: 15s, 직렬)
    → 자막 실패 시 최대 27s 소비 후 null 반환

[서버 /api/chat → LangGraph]
  ⑤ Router: gemini-2.5-flash-lite LLM 호출 (intent 분류, ~1~3s)
  ⑥ Generator: Gemini 요약 스트리밍
```

---

## 변경 1 — `src/hooks/useChatStream.ts` : 병렬 fetch

```ts
// Before: 직렬 (두 요청 시간의 합)
const metadata = await fetchUrlContent(url);
const transcript = await fetchYoutubeTranscript(videoId);

// After: 병렬 (가장 느린 쪽의 시간만)
const [metadata, transcript] = await Promise.all([
  fetchUrlContent(url),
  videoId ? fetchYoutubeTranscript(videoId) : Promise.resolve(null),
]);
```

- `fetchUrlContent`(메타데이터)와 `fetchYoutubeTranscript`(자막)는 순서 의존성 없음
- 예상 절감: **2~5s**

---

## 변경 2 — `api/_lib/agent/nodes/router.ts` : YouTube fast-path

```ts
// YouTube URL 감지 시 LLM 분류 없이 즉시 general 라우팅
const hasYoutubeUrl = /(?:youtube\.com\/|youtu\.be\/)/.test(textContent) ||
    (state.webContent && /(?:youtube\.com\/|youtu\.be\/)/.test(state.webContent));
if (hasYoutubeUrl && !hasMedicalKeyword) {
    console.log('[LangGraph] Router fast-path: YouTube URL → general');
    return { nextNode: "generator", intent: "general" };
}
```

- YouTube 요청은 항상 `general` intent → `gemini-2.5-flash-lite` 호출 불필요
- 기존 이미지 fast-path(`hasImage && !hasMedicalKeyword`)와 동일한 패턴으로 추가
- 의약품 키워드 동반 시 fast-path 제외 (기존 LLM 분류 유지)
- 예상 절감: **1~3s**

---

## 변경 3 — `api/fetch-transcript.ts` : 서버 timeout 단축

| 구간 | Before | After |
|------|--------|-------|
| YouTube 페이지 HTML fetch | 12s | 6s |
| caption XML fetch | 15s | 8s |
| **실패 시 최대 대기 합산** | **27s** | **14s** |

자막이 없는 영상, 자막 URL 접근 불가, YouTube 서버 지연 등 실패 케이스에서 사용자 대기 시간 단축.

---

## 변경 4 — `services/geminiService.ts` : 클라이언트 timeout 단축

| | Before | After |
|--|--------|-------|
| `fetchYoutubeTranscript` 클라이언트 timeout | 45s | 20s |

서버 최대가 14s로 줄었으므로 20s면 충분히 여유 확보.

---

## 케이스별 효과 요약

| 케이스 | 개선 전 | 개선 후 |
|--------|---------|---------|
| 자막 추출 **성공** (일반적인 경우) | 메타데이터 + 자막 시간의 합 | 둘 중 느린 쪽 시간만 |
| 자막 추출 **실패** (timeout 도달) | 최대 ~27s 대기 후 fallback | 최대 ~14s 대기 후 fallback |
| Router LLM 호출 | 매 요청마다 ~1~3s | 0s (즉시 라우팅) |

---

## 미개선 항목 (구조적 한계)

- **native video fallback 지연**: 자막 없을 때 Gemini가 영상 직접 다운로드+분석 — 영상 길이에 따라 30~60s 이상 소요. 구조상 제거 불가, 사용자 안내 메시지로 대응 중.
- **트랜스크립트 캐싱 없음**: 동일 영상 반복 요청 시 매번 re-fetch. 세션 내 `lastActiveDoc.extractedText`로 부분 완화되어 있음.

# 세션 타이틀 생성 2차 개선 (2026-04-30 — 완료)

## 배경

1차 개선(5단어→10단어) 후에도 "앤트로픽, IPO 앞두고" 같이 미완성 구로 끝나는 제목이 생성됨.

## 원인 분석

두 가지 복합 원인:

1. **`\n\n제목:` 접미사**: 프롬프트가 `제목:`으로 끝나면 모델이 신문 헤드라인 단편 스타일로 완성 — `앞두고`, `관련` 등 미완성 구로 종료.
2. **어시스턴트 응답 마크다운 전체 전달**: 불릿·볼드·코드블록 등 서식 노이즈로 인해 모델이 응답 첫 구절만 제목으로 추출.

## 변경

### `api/summarize-title.ts`

**히스토리 전처리 추가**

```ts
// Before: 마크다운 포함 전체 전달
history.slice(-6).map((m) => `${role}: ${m.content}`).join("\n")

// After: 어시스턴트 응답 마크다운 제거 + 300자 트림
const stripMarkdown = (text) =>
    text.replace(/```[\s\S]*?```/g, '').replace(/[*_`>#~\[\]]/g, '').replace(/\s+/g, ' ').trim();

history.slice(-6).map((m) => {
    if (m.role === 'user') return `User: ${m.content}`;
    const plain = stripMarkdown(m.content || '');
    return `Assistant: ${plain.slice(0, 300)}${plain.length > 300 ? '...' : ''}`;
}).join("\n");
```

**프롬프트 개선**

| 항목 | Before | After |
|------|--------|-------|
| 미완성 구 금지 | 없음 | `'~앞두고', '~관련' 같은 미완성 구 금지` 명시 |
| 단어 수 | "10단어 이내" | "5~15단어 사이" |
| 완결성 강제 | 없음 | "완결된 명사형 제목" 명시 |

**`제목:` 접미사 제거**

```ts
// Before
`${TITLE_PROMPT}\n\n[대화 내용]\n${chatHistoryText}\n\n제목:`

// After
`${TITLE_PROMPT}\n\n[대화 내용]\n${chatHistoryText}`
```

## 효과

- 개선 전: "앤트로픽, IPO 앞두고"
- 개선 후: "앤트로픽 IPO 앞두고 9000억 달러 기업가치로 신규 투자 유치 검토"

# 세션 타이틀 생성 3차 개선 (2026-04-30 — 완료)

## 원인

`gemini-2.5-flash`는 기본적으로 **thinking(내부 추론) 활성화** 상태.  
Thinking 토큰이 `maxOutputTokens` 예산을 먼저 소비하기 때문에 실제 제목 출력에 남는 토큰이 부족해 단어 중간에서 잘림.

예: maxOutputTokens 400 중 thinking이 380 소비 → 제목 출력에 20토큰만 남음 → "줄기세" 에서 컷

## 변경

### `api/summarize-title.ts`

```ts
config: {
    temperature: 0.3,
    maxOutputTokens: 400,
    thinkingConfig: { thinkingBudget: 0 },  // thinking 비활성화
}
```

제목 생성은 단순 작업 — thinking 불필요. `thinkingBudget: 0` 으로 비활성화 시 400토큰 전부 제목 출력에 사용.

| 어시스턴트 컨텍스트 | 300자 | 500자 |

## 효과

- 개선 전: "김광수 교수, 파킨슨병 환자 맞춤 줄기세" (단어 중간 잘림)

# YouTube Native Video 분석 타임아웃 수정 (2026-04-30 — 완료)

## 배경

transcript 추출 제거 후 모든 YouTube 요약이 native video 분석으로 전환됐지만, 한강 다큐멘터리 같은 장편 영상에서 응답이 잘리는 현상 발생.

## 원인

Vercel 로그 분석 결과:
```
Gemini API 호출 시간: 55,576ms
Vercel maxDuration: 60,000ms
남은 여유: ~4.5s → 초과 → 응답 잘림
```

`gemini-2.5-flash` 기본 thinking(무제한)이 20~30s를 소비.  
영상 다운로드 + 처리 + thinking + 응답 생성 합산이 60s를 초과.

## 변경

### `api/_lib/agent/nodes/generator.ts`

YouTube native video 분석 시(`isYoutubeRequest && hasVideoData`) `thinkingBudget: 1024` 적용:

```ts
const thinkingConfig = (isYoutubeRequest && hasVideoData)
    ? { thinkingBudget: 1024 }
    : undefined;

config: {
    ...
    ...(thinkingConfig ? { thinkingConfig } : {}),
}
```

스트리밍 경로 및 fallback 경로 모두 적용.

## 병행 작업 — transcript 추출 전면 제거

모든 server-side transcript 방법이 YouTube IP 차단으로 실패 확인:
- HTML 스크래핑 (`/watch`): Vercel iad1, hnd1 모두 60s hang
- timedtext API: 동일 차단
- `youtubei.js` InnerTube API: HTTP 400 + 백그라운드 hang

**제거 내용**:
- `fetchYoutubeTranscript()` 함수 제거 (`geminiService.ts`)
- YouTube 플로우에서 transcript fetch 제거, 바로 native video로 진입 (`useChatStream.ts`)
- `youtubei.js` 패키지 언인스톨 (~1MB 번들 절감)
- `fetch-transcript.ts` → 즉시 에러 반환 (60s hang 완전 제거)

## 효과

| 항목 | 이전 | 이후 |
|------|------|------|
| Gemini API 시간 | 55.5s (타임아웃) | ~35-40s (정상 완료) |
| transcript 대기 | 5s (항상 실패) | 0s (제거) |
| 응답 완성도 | 잘림 | 타임스탬프 포함 완전한 요약 |

# YouTube Native Video thinkingBudget 0으로 하향 (2026-04-30 — 완료)

## 배경

`thinkingBudget: 1024` 적용 후에도 긴 영상에서 60s 타임아웃 재발.

## 원인

```
13:30:43.308  함수 시작 (60s 카운트)
13:30:43.620  첫 번째 키 429 즉시 실패 → 0.3s 낭비
13:30:43.621  두 번째 키로 재시도 (남은 시간 ~59.7s)
13:31:43.308  타임아웃
```

긴 영상(15분+)은 Gemini 영상 다운로드·처리에만 30~50s 소요.  
여기에 `thinkingBudget: 1024` (~10s) + 응답 생성이 더해지면 60s 초과.  
YouTube 요약은 deep reasoning이 불필요한 작업이므로 thinking 완전 비활성화가 적합.

## 변경

### `api/_lib/agent/nodes/generator.ts`

```ts
// Before
{ thinkingBudget: 1024 }

// After
{ thinkingBudget: 0 }  // 완전 비활성화
```
