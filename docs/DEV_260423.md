## 버그 분석: 이미지 분석 시 간헐적 "Load failed" alert

### 증상
이미지 분석 요청 시 가끔 "Load failed" (또는 "업로드 실패") alert가 뜨며 분석 실패.
재시도하면 정상 동작.

### 원인 분석 (복합)

**원인 1 — `api/chat.ts` bodyParser 한계 (가장 유력)**

```
// useChatStream.ts:121
if (!isVideo && estimatedSize < (3 * 1024 * 1024) && isBase64) {
  finalAttachments.push(attachment);  // 업로드 안 하고 base64 인라인 전송
}
```

- `estimatedSize = base64Length * 0.75` (바이너리 추정치)
- 3MB 이미지 → JSON body 내 base64 문자열은 **~4MB**
- `api/chat.ts`에 `export const config` 없음 → Vercel 기본 bodyParser 한계 **4.5MB**
- 여기에 history까지 더해지면 4.5MB 초과 → Vercel이 함수 실행 전 연결 거부
- `fetch()` 단에서 실패 → Safari: `TypeError: Load failed`, Chrome: `TypeError: Failed to fetch`

**원인 2 — Vercel Cold Start 지연**

- 콜드스타트 시 LangGraph 초기화 → Router Node → Vision Node → Gemini API 호출까지 지연
- 첫 heartbeat(8초) 이전에 클라이언트/모바일 프록시가 연결 종료
- 재시도 시 함수가 워밍 상태 → 정상 동작

**원인 3 — 모바일 네트워크 순간 단절**

- 대용량 base64 바디 전송 중 네트워크 순간 불안정 → `fetch()` 실패

### 재시도가 되는 이유
- Cold start 이후 함수 워밍 → 두 번째 요청 빠르게 처리
- 네트워크 순간 단절은 일시적
- 4.5MB 경계선 근처 크기의 이미지는 retry 시 body 조합이 달라지면 통과되기도 함

### 수정 방향
- [ ] **I1**: `api/chat.ts`에 `export const config = { api: { bodyParser: { sizeLimit: '10mb' } } }` 추가 (즉각 효과)
- [ ] **I2**: `useChatStream.ts` inline 임계값 3MB → 1MB로 낮추기 — 이미지는 Supabase storage 업로드 후 URL로 전달
- [ ] **I3**: `vision.ts` Vision Node에 Gemini API 타임아웃 설정 (Cold start 대응)

## 버그 분석: 멀티턴 이미지 분석 후 응답 품질 저하

### 증상
이미지 분석(Turn 1) 이후 비교·요약 질의(Turn N)를 하면 응답이 짧거나 이미지 내용이 누락됨.
Turn 1 바로 다음(Turn 2)보다 대화가 더 진행된 후에 더 두드러짐.

### 원인 분석

**원인 1 — `isRecent` 컷오프로 이미지가 history에서 누락 (핵심)**

`api/chat.ts:49`:
```ts
const isRecent = index >= array.length - 3;
// 마지막 3개 메시지에 해당하지 않으면 이미지 제거 → [Attached File: image/jpeg] 텍스트로 대체
```

| 상황 | history 길이 | Turn 1 이미지 포함 여부 |
|------|------------|----------------------|
| Turn 2 (바로 다음) | 2 | ✅ 포함 (index >= -1) |
| Turn 3 이후 | 4+ | ❌ 누락 (index 0 < array.length-3) |

이미지 업로드 후 **2번 이상 추가 대화** 후에 질의하면 이미지가 history에서 탈락.

**원인 2 — 이미지 누락 시 Google Search 강제 활성화**

`generator.ts:151`:
```ts
let useGoogleSearch = !hasMultimodalContent;
// 이미지가 history에서 빠지면 hasMultimodalContent = false → Google Search ON
```

이미지가 누락되면:
1. `hasMultimodalContent = false`
2. `useGoogleSearch = true` 자동 전환
3. 모델이 대화 컨텍스트 대신 Google 검색 스니펫을 기반으로 응답
4. 결과: 이미지와 무관한 짧고 generic한 응답

**원인 3 — `contextInfo` (Vision 추출 결과) 미영속**

Vision Node가 Turn 1에서 추출한 구조화 데이터(`contextInfo`)는 LangGraph 그래프 state에만 존재.
Turn 2부터는 이 정보가 없어 모델이 Turn 1 AI 응답 텍스트에만 의존.

**원인 4 — base64 이미지가 history에 남아 body size 압박**

Turn 1 이미지가 inline base64(~4MB)로 세션에 저장된 경우,
Turn N 요청 body = base64 이미지 + 누적 대화 history → 4.5MB 한계 근접 → 파싱 품질 저하.

### 발생 흐름

```
Turn 1: 이미지 업로드 + 분석 요청
  → Vision Node: 구조화 추출 → contextInfo 주입
  → Generator: 이미지 직접 참조 + contextInfo → 풍부한 응답 ✅

Turn 2 (바로 다음): 비교/요약 질의
  → isRecent = true (history 2개) → 이미지 포함
  → contextInfo 없지만 이미지는 있음 → 무난하게 동작 ⚠️

Turn 3+: 비교/요약 질의
  → isRecent = false (Turn 1 이미지 탈락)
  → hasMultimodalContent = false → Google Search 활성화
  → 검색 스니펫 기반 짧은 응답 ❌
```

### 해결 방안

**방안 A — `isRecent` 기준 완화 (즉각 적용, 가장 효과적)**

`api/chat.ts:49` 수정:
```ts
// Before
const isRecent = index >= array.length - 3;

// After: 이미지가 있는 메시지는 항상 포함, 나머지는 기존 기준 유지
const msgHasImage = msgAttachments.some((a: any) => a.mimeType?.startsWith('image/'));
const isRecent = msgHasImage || index >= array.length - 3;
```
- 효과: 이미지 포함 메시지는 history에서 절대 누락되지 않음
- 비용: 오래된 이미지가 계속 context에 포함되어 body 크기 증가 (→ 방안 B와 함께 적용 필요)

**방안 B — inline 임계값 축소로 body 크기 감소 (I2와 동일)**

`useChatStream.ts:121` 수정:
```ts
// Before
if (!isVideo && estimatedSize < (3 * 1024 * 1024) && isBase64) {

// After
if (!isVideo && estimatedSize < (1 * 1024 * 1024) && isBase64) {
```
- 1MB 초과 이미지는 Supabase storage URL로 전환
- history에 남는 데이터가 base64 대신 URL → body 크기 대폭 감소
- 방안 A와 함께 적용 시 이미지 history 유지 + body 경량화 동시 달성

**방안 C — `useGoogleSearch` 판단 시 history 이미지 여부 포함**

`generator.ts:151` 수정:
```ts
// 현재: 현재 턴 메시지의 multimodal 여부만 체크
let useGoogleSearch = !hasMultimodalContent;

// 수정: history 메시지의 이미지 유무도 포함하여 판단
const historyHasImage = state.messages.some((m: any) =>
    Array.isArray(m.content) && m.content.some((p: any) =>
        p.type === 'image_url' || p.inlineData || (p.fileData && !p.fileData.fileUri?.includes('youtube'))
    )
);
let useGoogleSearch = !hasMultimodalContent && !historyHasImage;
```
- 효과: 이미지 분석 대화 중 Google Search가 불필요하게 활성화되는 것 방지
- 방안 A로 이미지를 history에 유지하면 자동으로 커버되지만, 안전망으로 추가

**방안 D — Vision contextInfo 세션 저장 (중장기)**

Turn 1 Vision Node 결과(`contextInfo`)를 Supabase `chat_sessions` 테이블의 메타데이터 컬럼에 저장.
이후 Turn N 요청 시 해당 세션의 contextInfo를 불러와 system prompt에 주입.
- 효과: 몇 턴이 지나도 약 식별 결과 등 구조화 데이터 유지
- 비용: DB 스키마 변경 + API 수정 필요

### 적용 우선순위

| 순위 | 방안 | 난이도 | 효과 |
|------|------|--------|------|
| 1 | **A + B 동시 적용** | 낮음 (코드 2줄) | 핵심 문제 해결 |
| 2 | **C** | 낮음 (코드 5줄) | Google Search 오염 방지 |
| 3 | **D** | 높음 (DB 스키마) | 완전한 contextInfo 영속 |

- [ ] **I4**: `api/chat.ts` — `isRecent` 조건에 이미지 메시지 예외 추가 (방안 A)
- [ ] **I5**: `useChatStream.ts` — inline 임계값 1MB로 축소 (방안 B, I2와 동일)
- [ ] **I6**: `generator.ts` — `useGoogleSearch` 판단 시 history 이미지 여부 포함 (방안 C)
- [ ] **I7**: Vision contextInfo DB 저장 + 멀티턴 주입 (방안 D, 중장기)

## 버그 분석: 약품 카드 이미지 일관적 누락

### 증상
약품 정보 카드에서 이미지가 항상(또는 자주) 표시되지 않고 "이미지 준비 중" 상태로 남음.

### 이미지 로딩 흐름

```
drug-info-tool → MFDS ITEM_IMAGE → LLM → json:drug.image_url
DrugRenderer → isSearchOrEntryPage 체크 → proxiedImageUrl 결정
             → useEffect: /api/sync-drug-image (Supabase 캐시)
             → 실패 시 /api/proxy-image 폴백
```

### 원인 분석

**원인 1 — `proxy-image.ts` content-type 미검증 (가장 직접적)**

`api/proxy-image.ts:85-92`:
```ts
const contentType = response.headers.get('content-type') || 'image/jpeg';
const buffer = await response.arrayBuffer();
res.setHeader('Content-Type', contentType);
return res.status(200).send(Buffer.from(buffer));  // HTML도 그냥 통과
```

nedrug.mfds.go.kr는 서버 점검 시 이미지 URL 요청에 HTML 페이지를 반환함.
`sync-drug-image.ts`는 content-type 체크 후 거부하지만, `proxy-image.ts`는 HTML을 그대로 통과시킴.
브라우저가 `<img src="/api/proxy-image?url=...">` 로 HTML을 받으면 → `onError` 발생 → `imageError = true` → "이미지 없음".

**원인 2 — MFDS `ITEM_IMAGE` null 시 폴백 없음**

`drug-info-tool.ts:400`:
```
공식 이미지URL: ${item.ITEM_IMAGE || 'null'}
```

MFDS DB에 이미지가 없는 약품은 `ITEM_IMAGE = null` → LLM이 `image_url: null` 세팅.
`DrugRenderer.useEffect`:
```ts
if (!isValidUrl(data.image_url)) {
    setSyncing(false);
    return;  // sync 자체를 시도하지 않음
}
```
`pharm_url`이나 `connectdi_url` 경유 이미지 시도 경로 없음 → 즉시 "이미지 없음".

**원인 3 — LLM이 ConnectDI search URL을 `image_url`로 혼용**

tool 출력에 `공식 이미지URL`과 `ConnectDI_URL` 두 URL이 나란히 존재.
LLM이 가끔 ConnectDI URL(`search_result` 포함)을 `image_url`에 세팅하면:

```ts
// DrugRenderer.tsx
const isSearchOrEntryPage =
    data.image_url?.includes('search_result') ||  // → true
    ...
const proxiedImageUrl = ... !isSearchOrEntryPage ? `/api/proxy-image?...` : null;
// → proxiedImageUrl = null (proxy 폴백 없음)
```

sync만 믿어야 하는데 ConnectDI 검색 결과 name-match score `< 40` 시 이미지 없음.

**원인 4 — sync 실패 시 `DrugRenderer`에서 무처리**

`DrugRenderer.tsx:224-236`:
```ts
if (response.ok) {
    setSyncedUrl(publicUrl);
}
// sync 실패(4xx/5xx) 시 아무것도 안 함, syncing만 false로
```

`isSearchOrEntryPage = true`인 URL에서 sync 실패 → `proxiedImageUrl = null` 유지 → "이미지 없음".
실패 원인도 콘솔에 남지 않음.

- ConnectDI는 MFDS와 **동일한 이미지 ID**를 사용 → 완벽한 폴백 가능
- ConnectDI 폴백이 **fetch 자체 실패(throw)** 시에는 실행되지 않음 (HTTP 에러 응답 시에만 실행)

### 실제 장애 흐름 (테스트로 확인)

```
1. DrugRenderer → /api/sync-drug-image (MFDS URL 전달)
2. sync: fetchWithRetry() → nedrug 차단 → throw (ECONNRESET/timeout)
3. sync catch 블록: resolveInflight(null), status 500 반환
   ↑ ConnectDI 폴백 미실행 (throw 경로에 폴백 없음)
4. DrugRenderer: response.ok = false → syncedUrl = null
5. proxiedImageUrl = /api/proxy-image?url=MFDS_URL (proxy 폴백)
6. proxy-image: MFDS fetch → 동일하게 차단 → 500 반환
7. <img onError> → imageError = true → "이미지 없음" ❌
```

**ConnectDI에 동일 이미지가 있음에도 폴백 미실행으로 누락.**

### 추가 발견: MFDS 검색 "밀리그람" 전략 누락
