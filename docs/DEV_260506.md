## 3. Intent 분류 설계

### 3-1. 분류 주체
- **gemini-2.5-flash-lite** (현재 router.ts L82)
- `temperature: 0`, JSON 응답 강제
- LLM 실패 시 → 키워드 heuristic fallback

### 3-2. 핵심 원칙 — "이동 의도 vs 정보 의도"

```
사용자가 실제로 이동해야 하는가?
    YES → pharmacy_search / hospital_search
    NO  → drug_info / medical_qa
```

### 3-3. 충돌 케이스 및 기준

| 메시지 예시 | 올바른 분류 | 판단 근거 |
|------------|-----------|---------|
| `"타이레놀 부작용"` | `drug_info` | 약 자체 정보 요청 |
| `"강남구 약국"` | `pharmacy_search` | 위치 탐색 |
| `"두통약 파는 약국 어디야"` | `pharmacy_search` | 이동 의도 우선 |
| `"약국에서 처방 없이 살 수 있는 약"` | `drug_info` | 약 정보 질의 |
| `"강남구 병원"` | `hospital_search` | 위치 탐색 |
| `"응급실 가야 할까?"` | `medical_qa` | 의학 판단 요청 |
| `"지금 열린 응급실 어디야"` | `hospital_search` | 이동 의도 |
| `"병원 드라마 추천"` | `general` | 엔터테인먼트 |
| `"내과 증상이 뭐야"` | `medical_qa` | 의학 지식 |
| `"내과 있는 병원 찾아줘"` | `hospital_search` | 이동 의도 |

### 3-4. LLM 프롬프트 설계 (의도 설명 방식)

기존 프롬프트 패턴 (keyword 나열) → **의도 + 예시 + NOT 패턴** 3요소 구성으로 변경

```
"drug_info"       : 약품 자체 정보 (성분·효능·부작용·복용법).
                    약국 위치 찾기가 아님.
                    Ex) "타이레놀 부작용", "두통약 추천", "아스피린 성분"
                    NOT: "약국 어디야"(→ pharmacy_search)

"pharmacy_search" : 사용자가 실제 약국에 방문하려는 의도.
                    지역명·영업시간·전화번호가 목적.
                    Ex) "강남구 약국", "지금 여는 약국", "두통약 파는 약국 어디야"
                    NOT: "두통약 성분"(→ drug_info)

"medical_qa"      : 증상·질병·치료법 등 의학 지식 질의.
                    병원을 찾는 것이 아닌 정보 요청.
                    Ex) "두통 원인", "혈압약 먹어도 돼?", "이 증상 뭐야"
                    NOT: "두통 병원 어디야"(→ hospital_search)

"hospital_search" : 사용자가 실제 병원에 방문하려는 의도.
                    지역명·진료시간·응급실 여부가 목적.
                    Ex) "강남구 병원", "응급실 어디야", "내과 병원 찾아줘"
                    NOT: "내과 가야 할 증상"(→ medical_qa)
                    NOT: "병원 드라마 추천"(→ general)
```

### 3-5. Fast-path 정책 (LLM 생략)
- YouTube URL → `general` 유지
- 이미지 + 비약품 → `general` 유지
- 약국·병원은 **fast-path 없음** — LLM 판단에 완전 위임
  - 이유: "약국"이라는 단어가 포함돼도 의도가 다양 (fast-path 시 오분류 위험)

| 필드 | API 필드 | 표시 방식 |
|------|---------|---------|
| 약국명 | `DUTYNAME` | 헤더 타이틀 |
| 주소 | `DUTYADDR` | 헤더 서브텍스트 |
| 영업상태 | 계산 (현재 KST) | 🟢 영업중 / 🔴 영업종료 배지 |

| 전 요일 + 공휴일 | `DUTYTIME1~8S/C` | 8칸 그리드 |
| 전화 | `DUTYTEL1` | 탭 클릭 → 전화 앱 연결 |
| 지도 링크 | `WGS84LAT/LON` | 카카오지도 링크 버튼 |

### 4-3. 병원 카드 포함 정보
| 필드 | API 필드 | 표시 방식 |
|------|---------|---------|
| 병원명 | `DUTYNAME` | 헤더 타이틀 |
| 병원종류 | `DUTYDIVNAM` | 배지 (종합병원/의원 등) |
| 응급실 | `DUTYERYN`, `DUTYEMCLSNAME` | 🚑 배지 |
| 주소 | `DUTYADDR` | 헤더 서브텍스트 |
| 진료상태 | 계산 (현재 KST) | 🟢 진료중 / 🔴 진료종료 배지 |
| 오늘 진료시간 | `DUTYTIME{요일}S/C` | 상단 강조 행 |
| 전 요일 + 공휴일 | `DUTYTIME1~8S/C` | 8칸 그리드 |
| 진료과목 | `DUTYINF` | 태그 목록 |
| 비고 | `DUTYETC` | 하단 안내 텍스트 |
| 전화 (일반) | `DUTYTEL1` | 탭 클릭 → 전화 앱 연결 |
| 전화 (응급) | `DUTYTEL3` | 별도 응급전화 버튼 |
| 지도 링크 | `WGS84LAT/LON` | 카카오지도 링크 버튼 |

### 4-4. 목록 모드 (검색 결과 여러 건)
- 약국/병원 모두 **목록형 + 단일 상세 카드** 2가지 모드
- 목록 아이템: 이름 / 주소 / 영업상태 배지 / 오늘 운영시간
- 페이지네이션: 5건씩 표시, 더보기 버튼

## 5. API 데이터 처리 주의사항

### 서울시 OpenAPI 공통
- 전체 건수: 약국 5,358건 / 병원 22,239건
- **페이지 전략**: 100건 로드 → 구 단위 클라이언트 필터
  - 구 지정 시: 500건 로드 (레퍼런스 패턴)
  - 구 미지정 시: 100건만 (전체 탐색 방지)
- **KST 기준 영업상태 계산** 필수 (서버 타임존 UTC 주의)
- 요일 인덱스 매핑: `DUTYTIME1S/C` = 월, `DUTYTIME7S/C` = 일, `DUTYTIME8S/C` = 공휴일

### GPS 좌표 활용
```
카카오지도 링크 포맷:
https://map.kakao.com/link/map/{이름},{위도},{경도}
```

### 진료과목 파싱 (`DUTYINF`)
- 자유형식 텍스트: `"내과, 외과, 정형외과, 신경과..."`
- 쉼표·`·`·줄바꿈으로 split → 태그 배열 변환

## 10. 전체 Intent 현황 및 개선 로드맵

### 10-1. 현재 Intent 전체 구조

| Intent | 트리거 조건 | 데이터 소스 | Tool | 출력 |
|--------|-----------|-----------|------|------|

| `chemistry` | 화학·분자구조 질의 | LLM | ❌ | `json:smiles` → ChemicalRenderer |
| `physics` | 물리 시뮬레이션 질의 | LLM | ❌ | PhysicsRenderer |
| `astronomy` | 천문·별자리 질의 | LLM | ❌ | ConstellationRenderer |
| `data_viz` | 데이터·통계·차트 질의 | LLM | ❌ | `json:chart` → ChartRenderer |
| `general` | 나머지 모든 것 | LLM + Google Search Grounding | ❌ | 마크다운 텍스트 |
| `pharmacy_search` | 약국 위치 탐색 (신규) | 서울시 약국 API | `PharmacyTool` (신규) | `json:pharmacy` → PharmacyRenderer |
| `hospital_search` | 병원 위치 탐색 (신규) | 서울시 병원 API | `HospitalTool` (신규) | `json:hospital` → HospitalRenderer |

### 10-2. `medical_qa` 흐름 (개선 완료)

```
사용자: "두통 원인이 뭐야?" / "혈압약 먹어도 되나?"
    ↓ router → medical_qa
    ↓ generator → SDK path (LangChain 미사용 유지)
       → Google Search Grounding 활성화 (이미지 없을 시 강제 ON)
       → thinkingBudget: 3000 (토큰 상한으로 레이턴시 최적화)
       → 검색 결과 기반 답변 생성
       → groundingChunks → 출처 칩 표시
    ↓ 마크다운 텍스트 + 출처 링크
```

#### (선택) PubMed 추가 연결 (로드맵)
- `medical_qa` + 임상/논문 관련 키워드 감지 시 PubMed Tool 병렬 호출
- Phase 2 `paper_search` 구현 후 연계 검토

### 10-3. `drug_id` 상세 흐름


## 11. PharmacyTool (약국 조회) 최적화 내역

### 11-1. LLM Zod 스키마 파싱 고도화
- **전국 API 대응**: 공공데이터포털 전국 약국 API의 요구사항에 맞춰 `sido`와 `sigungu` 파라미터 분리.
- **제한사항 회피**: '전주시 덕진구', '수원시 영통구' 등의 입력 시 API가 0건을 반환하는 버그 회피. LLM Prompt(Zod Description)에 "앞의 '시'를 제거하고 '덕진구', '영통구'만 전달하라"고 강력히 지시하여 해결.

### 11-2. 스트리밍 에러 (Failed to parse stream) 완전 해결
- **문제**: LLM이 방대한 약국 10~20개 데이터와 영업시간을 ````json` 텍스트로 생성하던 중, GoogleGenAI SDK의 스트림 청크 파싱 한계를 초과해 크래시 발생.
- **해결 1 (Token Minification)**: `JSON.stringify(..., null, 2)` 구조에서 들여쓰기와 줄바꿈을 전부 제거(Minify)하여 토큰 낭비 60% 이상 절감. 출력량은 10개로 유지.
- **해결 2 (Tool-side Generation)**: LLM이 툴 응답을 읽고 스스로 텍스트나 JSON을 다시 만들어내는 대신, Tool 내부에서 ````json:pharmacy ... ```` 마크다운 블록 문자열 자체를 반환하도록 설계.
- **해결 3 (Anti-Hallucination)**: LLM이 개원 연도 등을 지어내거나 일반 텍스트 목록으로 약국을 중복 나열하지 못하도록, 프롬프트에 `[지시사항]: 반환된 텍스트를 토씨 하나 틀리지 말고 그대로 출력하라`고 강제. (속도 대폭 향상 및 UI 카드 100% 렌더링 보장)

### 11-3. Direct Injection (Fast-pass) 최적화
- **문제**: LLM이 Tool의 결과물인 긴 JSON 데이터를 한 글자씩 스트리밍으로 뱉어내는 데만 추가로 약 10~15초 소요.
- **해결**: `chat.ts`에서 `PharmacyTool`의 결과물(마크다운 JSON 블록)이 반환되는 즉시(`on_tool_end` 이벤트), 이를 가로채 프론트엔드 SSE 스트림으로 **통째로 직접 전송(Direct Injection)**.
- **LLM 우회**: `generator.ts`에서 직전 메시지가 `PharmacyTool`의 결과물임을 감지하면, LLM의 텍스트 생성을 강제로 건너뛰고 빈 응답(`new AIMessage("")`)을 반환하도록 처리하여 10초 이상의 무의미한 LLM 낭비 시간 완전 소거.

### 11-4. 공공 API 병렬 호출 (Promise.allSettled)
- **문제**: 공공데이터포털 API의 느린 응답(회당 3~5초)을 `for`문으로 2회 순차 호출하여 최대 6~10초의 통신 병목 발생.
- **해결**: `Promise.allSettled`를 적용해 1페이지와 2페이지(총 200건)를 동시에(병렬) 요청하도록 수정하여 API 통신 대기 시간을 절반(3~5초)으로 단축. 한쪽 요청이 실패하더라도 다른 쪽의 데이터를 살리도록 안정성 강화.

### 11-5. LangChain 무한 재시도 늪 (maxRetries 0) 차단
- **문제**: API Key 한도 초과(429 Rate Limit) 발생 시, LangChain이 내부적으로 지수 백오프(Exponential Backoff)를 적용해 최대 6번씩 약 1분간 자체 재시도를 반복하며 Vercel 서버를 Hang(무응답) 상태로 만듦.
- **해결**: `generator.ts`의 `ChatGoogleGenerativeAI` 인스턴스 옵션에 `maxRetries: 0`을 명시적으로 부여. 429 에러 발생 시 LangChain 내부 재시도를 원천 차단하고 즉각 에러를 뱉어내게 하여, 우리가 자체 구현한 "Round Robin (다음 API 키 교체)" 로직이 0.1초 만에 동작하도록 수정.

### 11-6. 세부 키워드 필터링 (동 이름, 특정 약국명)
- **문제**: 공공데이터 API는 '시군구' 단위까지만 검색을 지원하여 "전주 중화산동 종로약국" 처럼 디테일한 검색 요청 시 한계 발생.
- **해결**: Zod 스키마에 `keyword` 파라미터를 추가하여 LLM이 세부 명칭을 추출하게 함. API에서 100~200건을 통째로 불러온 뒤, 클라이언트(Tool) 내부에서 `dutyAddr`(주소) 또는 `dutyName`(약국명)에 해당 `keyword`가 포함된 항목만 추려내는 텍스트 필터 로직 적용.

### 11-7. 검색 결과 0건 시 Google Search (search_web) 폴백
- **문제**: 세부 키워드 필터링을 거쳤거나 오타 등으로 인해 API 반환값이 0건일 때, 빈 화면이 출력되는 버그 발생 우려.
- **해결**: 결과가 0건일 시 에러 대신 `"[지시사항]: 제공된 search_web 툴을 사용하여 검색..."` 지시문을 반환하도록 수정. `generator.ts`에서 `pharmacy_search` 인텐트에도 `searchWebTool` 권한을 부여하여, 약국을 찾지 못하면 자연스럽게 DuckDuckGo/Google 웹 검색 결과를 요약해 주도록 안정성(Fallback) 확보.

### 11-8. PharmacyRenderer 모바일 UI(영업시간 그리드) 반응형 튜닝
- **문제**: 가로 폭이 좁은 모바일 기기에서 8칸(월~일, 공휴일)의 영업시간을 한 줄에 나열할 때, 공간 부족으로 시간 텍스트("09:00")가 강제 줄바꿈되며 레이아웃이 기괴하게 깨지는 현상 발생.
- **해결**:
  - `tracking-tighter`(자간 축소), `whitespace-nowrap`(줄바꿈 차단) 속성을 적용해 시간이 무조건 한 줄에 나오도록 강제.
  - 모바일 한정으로 폰트 크기(`text-[7.5px]`), 줄간격(`leading-none`), 컨테이너 좌우 여백(`px-2`), 그리드 간격(`gap-0.5`)을 극한으로 줄여 좁은 화면에서도 표가 정상 표출되도록 레이아웃 고도화 완료.

### 11-9. 약국 API Pagination 한계 돌파 및 단일 호출 병합 (정확도 100%)
- **문제**: 지연 시간 단축을 위해 1, 2페이지(200건)만 병렬 호출 후 클라이언트 필터링을 진행했으나, '전주시'처럼 범위가 넓고 약국이 400개가 넘어가는 지역 검색 시 뒤쪽에 위치한 구(완산구 등)의 약국이 아예 탐색 대상에서 누락되는 현상 발생.
- **해결**: 공공데이터 API가 `numOfRows=1000`을 완벽 지원함을 확인. 기존의 `Promise.allSettled` 병렬 호출 구조를 과감히 폐기하고, 단 한 번의 호출로 1000개를 로드(`pageNo: 1, numOfRows: 1000`)하도록 최적화. 지연 시간 증가는 미미하면서도 대도시 단위 전체 약국을 누락 없이 가져오는 완벽한 정확도 확보.

### 11-10. 공공데이터 API 행정구역(sigungu) 띄어쓰기 인식 버그 우회
- **문제**: API `Q1` 파라미터에 "전주시 완산구" 처럼 띄어쓰기가 포함된 행정구역명을 전송하면 에러 없이 0건을 반환하는 공공데이터 서버 측 결함 발견.
- **해결**: LLM 파싱 단계에서는 유연성을 보장하되, 서버(`pharmacy-tool.ts`) 통신 직전에 `sigungu.split(/\s+/)`를 통해 무조건 스페이스바 뒤의 마지막 단어("완산구")만 추출하여 전송하도록 강력한 자체 Sanitize(정제) 방어막 구축.

### 11-11. 휴일지킴이약국(주말/공휴일) 가중치 정렬 시스템 도입
- **문제**: 영업 중인 약국 내에서 단순 가나다순으로 정렬되어, 주말이나 늦은 밤에 가치 있는 약국이 후순위로 밀림.
- **해결**: 3단계 가중치 부여 시스템(`score`) 도입.
  - `+100점`: 현재 영업 중 (최우선)
  - `+20점`: 일요일 또는 공휴일 영업 약국
  - `+10점`: 토요일 영업 약국
  - 이 점수를 바탕으로 내림차순 정렬하여, '현재 열려있으면서 주말에도 영업하는 든든한 약국'이 무조건 최상단에 뜨도록 극적인 UX 향상.

### 11-12. 전화번호 클릭 기기별 네이티브/토스트 분기 처리
- **문제**: 데스크톱(PC) 브라우저에서 `tel:` 링크 클릭 시 전화 앱이 없어 빈 창으로 이동하거나 에러가 발생. 헤더 우측의 `1 / 2` 표시도 모바일에서 줄바꿈되며 레이아웃 저해.
- **해결**: 
  - 헤더의 불필요한 `1 / 2` 텍스트 제거.
  - `PharmacyRenderer.tsx`에 `isMobile` 정규식 감지 도입.
  - **모바일**: 기존 동작 유지 (OS 순정 통화 다이얼러 팝업 즉시 실행)
  - **데스크톱**: `e.preventDefault()`로 링크 이동 차단 → 번호 클립보드 복사 → `dispatchEvent(new CustomEvent('custom-toast'))`를 통해 앱 공용 토스트를 호출하여 부드러운 피드백 제공.

### 11-13. 공용 Toast 컴포넌트 멀티라인 반응형 고도화
- **문제**: `Toast.tsx`가 한 줄 고정(`whitespace-nowrap`)으로 되어 있어, 전화번호 등 긴 메시지 입력 시 꼬리가 `...`으로 짤림.
- **해결**: `whitespace-pre-wrap break-words` 적용 및 둥근 사각형(`rounded-2xl`), 너비(`max-w-[320px]`) 확장을 통해 여러 줄의 메시지도 앱의 글래스모피즘 디자인에 맞게 예쁘게 렌더링되도록 코어 컴포넌트 업그레이드.
